## Intuition

A recommender system's core data is deceptively simple: a giant, mostly-empty table of users versus items, where each filled-in cell is some signal of preference — a star rating, a click, a purchase, a completed watch. The system's job is to fill in the empty cells: predict, for a user and an item they haven't interacted with, how much that user would like that item.

The key insight that makes this tractable is that user preferences aren't arbitrary — they're driven by a much smaller number of underlying *factors* (genre preference, price sensitivity, aesthetic style) that both users and items can be described along. If you can discover those latent factors, you can describe every user and every item as a short vector in that shared factor space, and predict preference as simply how well a user's vector aligns with an item's vector — collapsing a huge sparse matrix into two small, dense ones.

<Callout type="tip" title="The one-sentence idea">
  Recommendation is the problem of explaining a huge, sparse user-item interaction matrix with a small number of shared latent factors — once you have those factors as vectors, recommending is just measuring alignment between a user vector and an item vector.
</Callout>

## Core Problem Formulations

- **Collaborative filtering:** predict a user's preference for an item using patterns across *all* users' and items' interactions, without needing any explicit content description of the item itself — "users who behaved like you also liked this."
- **Content-based filtering:** predict preference using explicit features of the items (genre, text description, image embeddings) and/or the user's profile, matching items to users by feature similarity rather than by collective behavior.
- **Ranking:** most production recommenders care less about predicting an exact rating and more about ranking a candidate set of items correctly for a given user — a subtly different objective from minimizing rating prediction error, addressed by learning-to-rank losses (pairwise or listwise) rather than plain regression.
- **Cold start:** predicting for a user or item with little or no interaction history — the point where collaborative filtering's core assumption (learn from past interactions) structurally breaks down.

## Key Techniques and How They Connect

**Matrix factorization** is the canonical collaborative filtering technique. Let $R \in \mathbb{R}^{m \times n}$ be the (mostly missing) user-item rating matrix for $m$ users and $n$ items. Matrix factorization approximates $R$ as the product of two low-rank matrices:

$$
R \approx U V^\top, \quad U \in \mathbb{R}^{m \times k}, \ V \in \mathbb{R}^{n \times k}
$$

where $k \ll \min(m, n)$ is the number of latent factors, each row $u_i$ of $U$ is user $i$'s embedding, and each row $v_j$ of $V$ is item $j$'s embedding. The predicted rating is simply the dot product $\hat{r}_{ij} = u_i^\top v_j$. Parameters are learned by minimizing squared error only over the *observed* entries, plus regularization to prevent overfitting on sparse data:

$$
\min_{U, V} \sum_{(i,j) \in \text{observed}} \left( r_{ij} - u_i^\top v_j \right)^2 + \lambda \left( \lVert U \rVert_F^2 + \lVert V \rVert_F^2 \right)
$$

This is conceptually a supervised, regularized cousin of the Singular Value Decomposition — SVD factors a *complete* matrix exactly into orthogonal components, while recommender-style matrix factorization learns an approximate factorization of an *incomplete* matrix by gradient-based optimization over only the observed entries. See [Singular Value Decomposition](/docs/foundations/linear-algebra/svd) for the exact, complete-matrix case this generalizes from. In practice, factorization is trained with alternating least squares (alternately fixing $U$ and solving for $V$, then vice versa, each a convex least-squares problem) or with stochastic gradient descent over observed ratings directly.

**Content-based filtering** takes the opposite approach: instead of inferring latent factors purely from collective interaction patterns, it builds item representations directly from item features (text description embeddings, genre tags, image embeddings) and recommends items similar to ones a specific user has liked before, using only that user's own history. This sidesteps collaborative filtering's reliance on *other* users' data entirely, at the cost of never surfacing recommendations outside a user's already-expressed taste profile (a "filter bubble" effect).

**Hybrid systems** combine both signals — collaborative filtering for the bulk of well-covered users/items where behavioral data is dense, content-based signals as a fallback or auxiliary feature where behavioral data is sparse — because the two approaches fail in largely non-overlapping situations, echoing the same principle that motivates hybrid dense/sparse retrieval in search.

**Deep learning approaches** generalize matrix factorization's simple dot product into a learned neural function of user and item embeddings (or replace factorized embeddings entirely with representations derived from raw content, sequence-of-interactions models, or graph neural networks over the user-item interaction graph), capturing nonlinear interaction effects a plain dot product cannot.

## The Cold-Start Problem

Matrix factorization needs observed interactions to learn a user's or item's embedding at all — a brand-new user with zero interactions, or a brand-new item nobody has rated yet, has no row to factorize, so $u_i^\top v_j$ is undefined for them in any meaningful sense. This is the **cold-start problem**, and it comes in two flavors:

- **New user cold start:** no interaction history to infer a preference embedding from.
- **New item cold start:** no ratings/interactions yet to infer a content embedding from.

Common mitigations include: falling back to **content-based filtering** using explicit item/user features when behavioral data is absent; using **popularity-based or demographic-based defaults** until enough interactions accumulate; explicitly asking new users for onboarding preferences (genre picks, a short survey) to bootstrap an initial embedding; and **exploration strategies** (showing a diverse or exploratory slate of items to new users specifically to *generate* the interaction data needed to later personalize).

## Practical Example

```python
import numpy as np

def train_matrix_factorization(R_observed, mask, k=20, lr=0.01, reg=0.1, epochs=50):
    m, n = R_observed.shape
    U = np.random.normal(scale=0.1, size=(m, k))
    V = np.random.normal(scale=0.1, size=(n, k))

    users, items = np.nonzero(mask)  # only iterate over observed entries
    for _ in range(epochs):
        for i, j in zip(users, items):
            err = R_observed[i, j] - U[i] @ V[j]
            U[i] += lr * (err * V[j] - reg * U[i])
            V[j] += lr * (err * U[i] - reg * V[j])
    return U, V

# Predicted preference for user i, item j after training:
# score = U[i] @ V[j]
```
