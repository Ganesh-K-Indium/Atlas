## Intuition

Labeled data is expensive; raw, unlabeled data is nearly free. Self-supervised learning (SSL) resolves that mismatch by having the model generate its own supervisory signal directly from the structure of the unlabeled data itself — no human annotation required. The trick is to design a **pretext task** whose solution forces the model to learn something genuinely useful about the data's structure, even though the task itself is often not the thing you actually care about. Two broad families of pretext task have proven most effective in practice: making the model recognize that two different "views" of the same thing are the same (**contrastive learning**), and making the model reconstruct or predict a deliberately hidden part of the input (**predictive/generative pretext tasks**, e.g. masked prediction).

<Callout type="tip" title="One-sentence core idea">
  Self-supervised learning creates labels from the data itself — either by teaching a model that augmented views of the same instance should have similar representations while different instances should not (contrastive), or by hiding part of the input and training the model to predict it (predictive/generative) — producing representations useful far beyond the pretext task itself.
</Callout>

## Formal Definition / Theory

Formally, given an unlabeled dataset $\{x_i\}_{i=1}^n$, self-supervised learning defines a pretext task with an automatically-derivable target $y_i = f(x_i)$ (e.g., a masked token, a rotation angle, or "does this pair come from the same source image") and trains an encoder $g_\theta$ to solve it. The premise is that a good solution to the pretext task requires the encoder to capture semantically meaningful structure — object identity, syntax, texture, composition — which transfers to downstream tasks (classification, detection, retrieval) via fine-tuning or simple linear probing, often with far fewer labeled examples than training from scratch.

### Contrastive learning (SimCLR-style)

Contrastive methods start from the assumption that two different augmented views of the *same* underlying instance should map to nearby points in representation space, while views from *different* instances should map far apart. Given a minibatch of $N$ images, each image $x_k$ is augmented twice (random crop, color jitter, blur, etc.) to produce a positive pair $(x_k^{(1)}, x_k^{(2)})$. Both augmented views pass through an encoder $g_\theta$ (e.g. a ResNet) followed by a small projection head $h_\phi$, producing embeddings $z_k^{(1)} = h_\phi(g_\theta(x_k^{(1)}))$ and $z_k^{(2)} = h_\phi(g_\theta(x_k^{(2)}))$.

The **normalized temperature-scaled cross-entropy loss (NT-Xent)** treats every other augmented image in the batch (all $2(N-1)$ of them) as a negative for a given anchor, and trains the model to maximize agreement with its true positive against all negatives:

$$
\ell(i, j) = -\log \frac{\exp\left(\text{sim}(z_i, z_j) / \tau\right)}{\sum_{k=1}^{2N} \mathbb{1}_{[k \neq i]} \exp\left(\text{sim}(z_i, z_k) / \tau\right)}
$$

where $\text{sim}(u, v) = \frac{u^\top v}{\lVert u \rVert \lVert v \rVert}$ is cosine similarity and $\tau$ is a temperature hyperparameter controlling how sharply the loss penalizes near-miss negatives. This is exactly the softmax cross-entropy loss over a $2N$-way classification problem where the "correct class" is the true positive pair — the encoder is being trained as an implicit $2N$-way classifier with no labels other than "which sample was I augmented from."

Two properties are essential in practice: (1) **strong, information-destroying augmentations** — if augmentations are too weak, the positive pairs are trivially close and the model learns nothing useful; if they destroy task-relevant information (e.g. color for a color-classification downstream task), representations transfer poorly; (2) **large batch sizes or memory banks**, since more negatives per anchor make the discrimination task harder and the learned representations correspondingly richer — this is precisely why contrastive frameworks are typically trained with very large batches or auxiliary negative-sample queues.

### Predictive / generative pretext tasks (masked prediction)

Rather than contrasting instances against each other, predictive approaches corrupt part of a single instance and train the model to reconstruct or predict the missing part from context. The canonical formalization: split input $x$ into observed context $x_{\text{obs}}$ and masked target $x_{\text{mask}}$ (a random subset of tokens, patches, or spans), and train:

$$
\mathcal{L} = -\log p_\theta(x_{\text{mask}} \mid x_{\text{obs}})
$$

For discrete tokens (as in masked-language-modeling-style objectives) this is a cross-entropy loss over a vocabulary at each masked position; for continuous signals (as in masked-image-modeling-style objectives) it is often a reconstruction loss (e.g. mean squared error in pixel or feature space) at masked patch locations. Crucially, solving this prediction problem well requires the model to build an internal representation that captures the statistical dependencies between parts of the input — syntax and semantics for text, object structure and texture for images — which is precisely the representation that transfers to downstream tasks.

The core difference from contrastive learning is the *unit of comparison*: contrastive objectives operate **across instances** (is this the same underlying thing as that one?), whereas predictive/generative objectives operate **within a single instance** (given this part, what's the rest?). Some modern approaches (e.g. self-distillation style methods that predict a target network's output representation directly, without an explicit reconstruction target) sit conceptually between the two, avoiding both large negative-sample counts and explicit pixel/token reconstruction — but the same instance-internal-vs-across-instance distinction still roughly applies.

## Mechanism, step by step (SimCLR-style contrastive pipeline)

1. Sample a minibatch of $N$ unlabeled examples.
2. Apply two independent random augmentations to each example, producing $2N$ views (each original contributes exactly one positive pair).
3. Encode all $2N$ views with a shared-weight encoder + projection head to get embeddings $z_1, \dots, z_{2N}$.
4. For each embedding, compute cosine similarity against all other $2N - 1$ embeddings in the batch.
5. Apply the NT-Xent loss: pull the true positive pair's similarity up, push all other $2(N-1)$ pairs (negatives) down, scaled by temperature $\tau$.
6. Backpropagate through both the projection head and the encoder; after pretraining, **discard the projection head** and keep only the encoder — it is the encoder's representations, not the projection head's, that transfer well to downstream tasks (an empirical finding across contrastive SSL work).
7. For downstream use: either freeze the encoder and train a small linear classifier on top ("linear probing") or fine-tune the whole encoder on a labeled task with far fewer labels than would be needed training from scratch.

## Why this matters: connection to foundation models

Self-supervised pretraining is the mechanism that underlies most modern large-scale pretraining, in both language and vision. Predicting masked or next tokens from vast unlabeled text corpora is a predictive self-supervised objective at massive scale — it requires no human labels, only raw text, and the resulting representations transfer to a huge range of downstream language tasks via fine-tuning or in-context adaptation. Contrastive and masked-prediction objectives play an analogous role for vision encoders. The general pattern — cheap, automatically-derived supervision from unlabeled data, at a scale far beyond what human annotation could ever provide — is the primary reason foundation models can be pretrained on internet-scale data before any task-specific labels are introduced. (Specific architectures and training recipes vary considerably across models and evolve quickly, so treat this as the general mechanism rather than a claim about any particular current system.)

## Code Example

```python
import torch
import torch.nn.functional as F

def nt_xent_loss(z1, z2, temperature=0.5):
    """
    z1, z2: (N, d) embeddings of two augmented views of the same N examples.
    Returns the NT-Xent contrastive loss (SimCLR-style).
    """
    N = z1.shape[0]
    z = torch.cat([z1, z2], dim=0)                      # (2N, d)
    z = F.normalize(z, dim=1)
    sim = z @ z.T / temperature                          # (2N, 2N) cosine similarities

    # Mask out self-similarity
    mask = torch.eye(2 * N, dtype=torch.bool, device=z.device)
    sim.masked_fill_(mask, float("-inf"))

    # Positive pairs: i <-> i+N (and i+N <-> i)
    targets = torch.cat([torch.arange(N, 2 * N), torch.arange(0, N)]).to(z.device)

    return F.cross_entropy(sim, targets)

# Usage sketch:
# z1 = projection_head(encoder(augment(batch)))
# z2 = projection_head(encoder(augment(batch)))
# loss = nt_xent_loss(z1, z2)
```

```python
# Masked prediction pretext task sketch (conceptual, framework-agnostic)
import numpy as np

def mask_tokens(tokens, mask_prob=0.15, mask_id=-1):
    tokens = np.array(tokens)
    mask = np.random.rand(len(tokens)) < mask_prob
    labels = np.where(mask, tokens, -100)  # -100 = ignore in loss for unmasked positions
    inputs = np.where(mask, mask_id, tokens)
    return inputs, labels

# Train the model to predict `labels` from `inputs` at masked positions only,
# using cross-entropy over the vocabulary (or MSE over pixel/feature targets
# for continuous, image-style masked prediction).
```

## Complexity

| Approach | Time complexity (per step) | Space complexity | Notes |
| --- | --- | --- | --- |
| Contrastive (batch size $N$) | $O(N^2 d)$ for pairwise similarity matrix | $O(N^2)$ for similarity matrix, $O(Nd)$ for embeddings | Quality scales strongly with batch size / negative count, driving high memory demand |
| Masked prediction | $O(L)$ per sequence/image for forward pass ($L$ = tokens/patches), model-dependent | $O(Ld)$ for activations | No pairwise comparison needed; cost dominated by the base encoder architecture |

## Common Mistakes

- Confusing self-supervised learning with unsupervised learning in general — SSL still trains with a supervised-style loss (cross-entropy, reconstruction), the difference is that the labels are derived automatically from the data rather than provided by humans.
- Assuming stronger augmentations are always better for contrastive learning — augmentations that destroy information relevant to the downstream task (e.g. removing all color information when color is a discriminative downstream feature) hurt transfer performance even though they make the pretext task itself harder.
- Evaluating a pretrained encoder purely by its pretext-task loss (e.g. contrastive loss value) rather than downstream transfer performance (linear probe accuracy, fine-tuning results) — a lower pretext loss does not guarantee better representations.
- Using a small batch size with a contrastive objective and expecting SimCLR-level quality — the number of negatives per anchor scales with batch size, and too few negatives makes the discrimination task too easy to be useful.
- Treating the projection head's output as the representation to use downstream — empirically, the layer *before* the projection head (the encoder's own output) transfers better; the projection head is typically discarded after pretraining.
