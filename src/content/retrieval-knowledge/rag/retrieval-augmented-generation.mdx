## Intuition

An LLM's knowledge is frozen at training time and compressed into its weights — it can't know about your company's internal documents, yesterday's news, or anything too rare to have been memorized precisely. Retraining the model every time information changes is prohibitively expensive.

**Retrieval-Augmented Generation** sidesteps this: instead of relying purely on parametric memory (what's baked into the weights), it retrieves relevant text from an external knowledge source at query time and inserts it directly into the model's context, so generation is *grounded* in explicit, inspectable, updatable evidence rather than the model's possibly-outdated or hallucinated recollection.

<Callout type="tip" title="The one-sentence idea">
  RAG = retrieve relevant documents for the query, then generate an answer conditioned on both the query and the retrieved documents — turning "hope the model remembers this correctly" into "show the model the source and ask it to read."
</Callout>

**Indexing (offline, done once per corpus update):**
1. Split source documents into chunks (paragraphs, fixed token windows, or semantically-aware splits).
2. Embed each chunk into a dense vector using an embedding model.
3. Store vectors in a vector database, indexed for fast approximate nearest-neighbor search.

**Query time (online, per user request):**
1. Embed the incoming query with the same embedding model.
2. Retrieve the top-$k$ most similar chunks via vector similarity search (typically cosine similarity or inner product).
3. Optionally **rerank** the retrieved candidates with a more expensive cross-encoder model that jointly scores (query, chunk) pairs — retrieval favors recall and speed; reranking favors precision.
4. Concatenate the retrieved chunks with the original query into a prompt, and let the LLM generate a response conditioned on both.

## Why Vector Search, and What It's Actually Computing

Chunks are embedded so that semantically similar text ends up close together in vector space — this lets retrieval match on *meaning* rather than exact keyword overlap. Similarity is typically measured with cosine similarity:

$$
\text{sim}(q, d) = \frac{q \cdot d}{\lVert q \rVert \, \lVert d \rVert}
$$

Since brute-force comparison against millions of vectors is too slow for real-time queries, production vector databases use **approximate nearest neighbor (ANN)** indexes — most commonly HNSW (Hierarchical Navigable Small World graphs) or IVF (inverted file index with clustering) — trading a small amount of recall for orders-of-magnitude faster lookup.

## Dense vs. Sparse vs. Hybrid Retrieval

  ]}

Most production RAG systems converge on hybrid search precisely because dense and sparse retrieval fail on different, largely non-overlapping query types.

## Minimal Implementation Sketch

```python
from openai import OpenAI
import numpy as np

client = OpenAI()

def embed(texts: list[str]) -> np.ndarray:
    resp = client.embeddings.create(model="text-embedding-3-large", input=texts)
    return np.array([d.embedding for d in resp.data])

def retrieve(query: str, chunk_embeddings: np.ndarray, chunks: list[str], k: int = 5):
    q_emb = embed([query])[0]
    scores = chunk_embeddings @ q_emb / (
        np.linalg.norm(chunk_embeddings, axis=1) * np.linalg.norm(q_emb)
    )
    top_k = np.argsort(scores)[::-1][:k]
    return [chunks[i] for i in top_k]

def generate(query: str, retrieved_chunks: list[str]) -> str:
    context = "\n\n".join(retrieved_chunks)
    prompt = f"Context:\n{context}\n\nQuestion: {query}\n\nAnswer using only the context above."
    resp = client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}],
    )
    return resp.choices[0].message.content
```

## Beyond Naive RAG

Naive "embed, retrieve top-k, stuff into prompt" RAG has well-known failure modes, which is why production systems layer on additional techniques:

- **Query rewriting / decomposition:** rephrase or split complex queries before retrieval (e.g., "compare X and Y" becomes two separate retrieval queries).
- **Reranking:** apply a cross-encoder to precisely re-score the top-$N$ candidates from the initial (cheaper) retrieval pass.
- **Chunking strategy:** overly small chunks lose context; overly large chunks dilute relevance signal and waste context window — chunk size and overlap are genuine tuning parameters.
- **GraphRAG:** builds a knowledge graph over entities and relationships extracted from the corpus, enabling retrieval that follows explicit relational structure (e.g., multi-hop questions) rather than pure semantic similarity — see [GraphRAG & Knowledge Graphs](/docs/retrieval-knowledge/rag/graphrag-knowledge-graphs) (roadmap).
- **Agentic RAG:** lets the model iteratively decide whether to retrieve again, refine its query, or answer — instead of a single fixed retrieve-then-generate pass.
