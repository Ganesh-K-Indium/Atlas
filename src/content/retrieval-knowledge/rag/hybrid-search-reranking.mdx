## Intuition

Dense retrieval (embedding similarity) and sparse retrieval (BM25-style keyword matching) fail on almost disjoint sets of queries. Dense retrieval is strong at paraphrase and semantic similarity but can under-weight an exact rare token — a product SKU, an error code, an uncommon proper noun — because the embedding model was never specifically trained to guarantee exact-match priority for such tokens. Sparse retrieval is the mirror image: it nails exact term overlap but has no notion of meaning, so a query about "car" won't match a document that only says "automobile."

**Hybrid search** runs both retrieval methods and combines their outputs into a single ranked list, covering both failure modes at once. But dense similarity scores and BM25 scores live on completely different, non-comparable scales — you can't just add them. This is exactly the problem **reciprocal rank fusion (RRF)** solves: instead of combining scores, it combines *ranks*, which are always comparable regardless of the underlying scoring function.

Separately, even a good hybrid retrieval pass over a large corpus is optimizing for recall (don't miss anything relevant) more than precision (put the single best answer first) — because it has to run cheaply over the entire corpus. **Reranking** adds a second, much more expensive but much more accurate model that only scores the small top-$N$ candidate set the first pass already narrowed down, refining precision at the top of the list.

<Callout type="tip" title="The one-sentence idea">
  Retrieve cheaply and broadly (hybrid: sparse + dense) to maximize the chance the right document is *somewhere* in your candidate set, then rerank expensively and narrowly (cross-encoder) to make sure it's *at the top*.
</Callout>

This pattern exists because there's a fundamental tension between retrieval cost and retrieval accuracy that no single model resolves well:

- **Bi-encoders** (used for both sparse-style and dense embedding retrieval) encode the query and each document *independently* into fixed vectors, so document embeddings can be precomputed once and indexed. Similarity is then a cheap dot product or BM25 score lookup. This scales to millions/billions of documents but the independence assumption caps accuracy — the model never actually looks at the query and document together.
- **Cross-encoders** concatenate the query and a candidate document and pass both through a transformer *jointly*, letting attention directly model interactions between query tokens and document tokens. This is far more accurate at judging true relevance, but it cannot be precomputed — every (query, document) pair requires a fresh forward pass, making it computationally infeasible to run over an entire corpus per query.

The two-stage pattern resolves this by applying the cheap bi-encoder/BM25 stage to the whole corpus (optimizing recall) and the expensive cross-encoder stage only to a small top-$N$ shortlist (optimizing precision), getting most of the accuracy of cross-encoding at a small fraction of its cost.

## Reciprocal Rank Fusion (RRF), Precisely

Given multiple ranked lists over the same document set (e.g., one from BM25, one from dense retrieval), RRF assigns each document a fused score based purely on its rank position in each list — not its raw score:

$$
\text{RRF}(d) = \sum_{r \,\in\, R} \frac{1}{k + \text{rank}_r(d)}
$$

where `R` is the set of rankers (e.g. sparse and dense), `rank_r(d)` is the rank of document `d` in ranker `r`'s list (1-indexed; documents not present in a list are simply omitted from that ranker's sum term), and `k` is a small constant (commonly 60, per the original paper's empirical tuning) that dampens the influence of very high ranks so the fused score doesn't blow up disproportionately for rank 1 versus rank 2.

Documents are then re-sorted by their fused `RRF(d)` score, descending. A few properties make RRF attractive as a fusion method specifically:

- **Scale-free.** It never touches the raw scores from either ranker, so it sidesteps the problem of BM25 scores and cosine similarities living on incomparable numeric ranges — a document ranked 1st by BM25 and one ranked 1st by dense retrieval contribute identically, regardless of the underlying score magnitudes.
- **Parameter-light.** The only real knob is `k`, and results are known to be fairly insensitive to its exact value across a reasonable range.
- **No training required.** Unlike learned fusion (e.g. training a small model to combine scores), RRF requires no labeled relevance data — it's a fixed formula, which makes it a strong, robust default before investing in a learned re-ranking layer.

### Worked example

Suppose a document `d` is ranked 2nd by BM25 and 5th by dense retrieval, with `k = 60`:

$$
\text{RRF}(d) = \frac{1}{60 + 2} + \frac{1}{60 + 5} = \frac{1}{62} + \frac{1}{65} \approx 0.0316
$$

A document ranked 1st by only one ranker and absent from the other's top results (say, rank 1 in BM25, not present in dense retrieval's list at all) would score just `1/61 ≈ 0.0164` — lower than `d` above, illustrating how RRF naturally rewards documents that multiple rankers agree are at least moderately relevant, over documents that only one ranker loves.

## Cross-Encoder Reranking, Precisely

A cross-encoder reranker is typically a transformer (often BERT-style) fine-tuned specifically for relevance scoring, taking the concatenated input `[CLS] query [SEP] document [SEP]` and producing a single scalar relevance score from the pooled output (commonly the `[CLS]` token representation passed through a small classification head). Because query and document tokens attend to each other directly within the same forward pass, the model can capture fine-grained interactions — negation, entity disambiguation, numerical comparison — that two independently-computed embeddings structurally cannot.

The trade-off is cost: a cross-encoder forward pass costs roughly as much as encoding a full sequence of `query + document` tokens, for every candidate, every query — there's no way to precompute or cache document-side representations the way bi-encoders allow. This is why cross-encoders are applied only to a shortlist (commonly $N = 20$ to $100$ candidates) rather than the full corpus.

$$
\text{score}_{\text{CE}}(q, d) = \text{Linear}\big(\text{Transformer}([\text{CLS}]\, q \,[\text{SEP}]\, d \,[\text{SEP}])_{\text{CLS}}\big)
$$

## Implementation Sketch

```python
from collections import defaultdict
from sentence_transformers import CrossEncoder

def reciprocal_rank_fusion(
    ranked_lists: list[list[str]], k: int = 60
) -> list[tuple[str, float]]:
    """ranked_lists: e.g. [bm25_doc_ids_in_rank_order, dense_doc_ids_in_rank_order]"""
    scores: dict[str, float] = defaultdict(float)
    for ranked_list in ranked_lists:
        for rank, doc_id in enumerate(ranked_list, start=1):
            scores[doc_id] += 1.0 / (k + rank)
    return sorted(scores.items(), key=lambda item: item[1], reverse=True)

def rerank(query: str, candidates: list[str], top_n: int = 10) -> list[tuple[str, float]]:
    model = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")
    pairs = [[query, doc] for doc in candidates]
    scores = model.predict(pairs)  # single forward pass per (query, doc) pair
    ranked = sorted(zip(candidates, scores), key=lambda x: x[1], reverse=True)
    return ranked[:top_n]

def hybrid_retrieve_and_rerank(
    query: str,
    bm25_results: list[str],
    dense_results: list[str],
    doc_lookup: dict[str, str],
    fusion_top_n: int = 50,
    final_k: int = 10,
):
    fused = reciprocal_rank_fusion([bm25_results, dense_results])
    shortlist_ids = [doc_id for doc_id, _ in fused[:fusion_top_n]]
    shortlist_texts = [doc_lookup[doc_id] for doc_id in shortlist_ids]
    return rerank(query, shortlist_texts, top_n=final_k)
```

    "The two-stage pattern lets you scale cheap retrieval to the full corpus while reserving expensive computation for a small, high-value shortlist.",

  cons={[
