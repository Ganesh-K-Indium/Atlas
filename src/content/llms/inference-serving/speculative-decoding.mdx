## Intuition

Autoregressive decoding generates one token per forward pass through the full model, and that forward pass is memory-bandwidth bound: streaming the weights (and KV cache) from HBM dominates the time, while the actual arithmetic for one token is cheap relative to the GPU's compute capacity. That means a single large-model forward pass has *slack* — it could verify many candidate tokens in parallel for roughly the same wall-clock cost as verifying one, because the bottleneck is memory traffic, not compute.

Speculative decoding exploits that slack: use a small, fast "draft" model to *guess* several tokens ahead cheaply, then hand all of those guesses to the large "target" model in a *single* parallel forward pass, which checks them all at once and accepts however many of the guesses turn out to agree with what the target model would have generated on its own. The clever part is the acceptance rule — it's designed so that, whatever fraction of guesses get accepted or rejected, the final output token's probability distribution is mathematically identical to sampling directly from the target model alone. Speedup comes from doing more useful work (verifying multiple tokens) per expensive (large-model) forward pass, not from any relaxation of output quality.

<Callout type="tip" title="The one-sentence idea">
  A cheap draft model proposes a few tokens ahead; the expensive target model verifies all of them in one parallel pass, accepting the longest correct prefix and resampling only from the first point of disagreement — this amortizes the target model's cost across multiple accepted tokens while leaving its output distribution exactly unchanged.
</Callout>

## Why This Preserves the Exact Target Distribution

Let $p(x)$ be the target model's distribution over the next token and $q(x)$ the draft model's distribution, both conditioned on the same prefix. The draft proposes a token $x \sim q$. The verification rule:

$$
\text{Accept } x \text{ with probability } \min\!\left(1, \frac{p(x)}{q(x)}\right)
$$

If $x$ is rejected, resample a token from the **residual distribution**:

$$
p_{\text{residual}}(x) = \frac{\max(0,\ p(x) - q(x))}{\sum_{x'} \max(0,\ p(x') - q(x'))}
$$

This is a modified-rejection-sampling scheme: it can be shown that the marginal distribution of the accepted-or-resampled token is *exactly* $p(x)$ — the target model's own distribution — regardless of how good or bad the draft model is. A better draft model only changes the *acceptance rate* (how often draft tokens survive verification, and therefore how much speedup you get); it never changes the distribution of the final output. This guarantee is what makes speculative decoding safe to use in production without altering generation quality: it's a pure latency optimization, not an approximation.

## The Mechanism, Step by Step

1. **Draft phase.** Starting from the current context, the small draft model autoregressively generates $K$ candidate tokens (e.g., $K = 4$–$8$), each drafted using its own (cheap) forward pass. This is fast because the draft model is much smaller than the target.
2. **Verification phase.** The target model runs a **single forward pass** over the original context plus all $K$ drafted tokens simultaneously (this is possible because, given the tokens, computing each position's logits in parallel is exactly what a Transformer does during training-style teacher forcing — no autoregression is needed for a fixed sequence). This produces, for every drafted position, the target model's true conditional distribution $p(x_i \mid \text{context}, x_{<i})$.
3. **Accept/reject, left to right.** Walk through the $K$ drafted tokens in order. For each, apply the acceptance test comparing $p(x_i)$ and $q(x_i)$. Accept tokens as long as the test passes; stop at the **first rejection**.
4. **Resample at the rejection point.** At the first token that fails the test (or after all $K$ are accepted), sample one new token from either the residual distribution $p_{\text{residual}}$ (if a draft token was rejected) or directly from $p$ (if all $K$ drafts were accepted, in which case a "bonus" token is sampled to make use of the target model's already-computed extra logits).
5. **Advance and repeat.** The accepted prefix (however long — anywhere from 1 to $K+1$ tokens) is appended to the context, the KV caches of both models are updated accordingly, and the process repeats from step 1.

Because step 2 verifies all $K$ tokens in one target-model forward pass, the *number of expensive target-model passes* needed to generate $n$ tokens drops from $n$ (one per token, standard decoding) to roughly $n / (\text{expected accepted length})$ — the speedup scales directly with how often the draft model's guesses agree with what the target model would have produced.

## Code Sketch

```python
# Simplified speculative decoding loop (conceptual — real implementations
# handle batching, KV cache management, and vectorized accept/reject checks).

def speculative_decode_step(target_model, draft_model, context, k=4):
    # 1. Draft: cheap model proposes k tokens autoregressively
    draft_tokens, draft_probs = [], []
    ctx = context
    for _ in range(k):
        q_dist = draft_model.next_token_dist(ctx)     # q(x | ctx)
        x = sample(q_dist)
        draft_tokens.append(x)
        draft_probs.append(q_dist[x])
        ctx = ctx + [x]

    # 2. Verify: ONE forward pass over context + all k drafted tokens
    target_dists = target_model.next_token_dists_parallel(context, draft_tokens)
    # target_dists[i] = p(x | context, draft_tokens[:i]) for i in 0..k

    accepted = []
    for i, x in enumerate(draft_tokens):
        p_x = target_dists[i][x]
        q_x = draft_probs[i]
        accept_prob = min(1.0, p_x / q_x)
        if random.random() < accept_prob:
            accepted.append(x)
        else:
            # Resample from the residual distribution p - q (clipped, renormalized)
            residual = clip_and_normalize(target_dists[i] - draft_probs_dist(i))
            new_token = sample(residual)
            return accepted + [new_token]

    # All k drafts accepted — sample one bonus token from the target's
    # already-computed extra logits (position k+1), for free.
    bonus_token = sample(target_dists[k])
    return accepted + [bonus_token]
```

## Complexity and Memory Analysis

| Quantity | Standard decoding | Speculative decoding |
| --- | --- | --- |
| Target-model forward passes for $n$ tokens | $n$ | $\approx n / (\text{expected accepted length})$ |
| Extra compute per verification pass | — | $O(K)$ tokens processed in parallel (cheap, since target pass is memory-bound anyway) |
| Extra models to host in memory | None | Draft model (small) + target model (large) |
| Output distribution | Exactly $p$ (target model) | Exactly $p$ (target model) — provably identical |
| Best case | 1x (baseline) | Speedup bounded by draft/target agreement rate and $K$ |

## Common Mistakes

<ProsCons
  pros={[
    "Mathematically guarantees the exact same output distribution as standard sampling from the target model — a genuine free lunch on latency, not a quality/speed trade-off.",
    "Exploits underused compute headroom in memory-bound decoding — the target model's forward pass was already 'wasting' compute capacity relative to its memory traffic.",
    "Composable with other techniques (quantization, PagedAttention-based serving) since it doesn't change the model architecture, just the decoding loop.",
  ]}
  cons={[
    "Requires hosting and maintaining a second (draft) model, adding engineering complexity and some fixed memory/compute overhead.",
    "Speedup is workload-dependent — it can be small or even negative if the draft model's predictions rarely agree with the target model's.",
    "Batching multiple concurrent requests, each with different accepted lengths per round, complicates scheduling relative to standard fixed-shape decoding — real serving systems need specialized kernels/schedulers to realize the theoretical speedup.",
  ]}
/>

- Believing speculative decoding trades quality for speed — the acceptance/resampling rule is specifically constructed so it doesn't; any implementation that skips proper rejection sampling (e.g., just accepting drafts by simple equality) breaks this guarantee.
- Assuming a bigger/better draft model is always better — a draft model that's nearly as expensive as the target model erodes the latency win, since drafting cost matters too.
- Forgetting that this optimizes **latency per request**, primarily useful in low-batch-size or interactive settings — at very large batch sizes the target model's forward pass may already be compute-bound, shrinking the "free" verification capacity this technique relies on.
