## Intuition

The standard attention formula looks compute-bound: $\text{softmax}(QK^\top/\sqrt{d})V$ involves matrix multiplies. But on a modern GPU, the bottleneck is rarely arithmetic — it's *moving data*. GPUs have a small amount of extremely fast on-chip memory (SRAM, tens of MB) and a large amount of much slower off-chip memory (HBM, tens of GB). A naive attention implementation computes the full $n \times n$ score matrix $QK^\top$, writes it out to HBM, reads it back to apply softmax, writes the result back to HBM, reads it again to multiply by $V$ — repeatedly shuttling an $O(n^2)$ array between slow and fast memory. For long sequences this HBM traffic, not FLOPs, is what makes attention slow.

FlashAttention's insight: restructure the *same* mathematical computation so the $n \times n$ matrix is **never fully written to HBM**. Instead, tile $Q$, $K$, $V$ into blocks small enough to fit in SRAM, and compute attention block-by-block, incrementally updating a running (unnormalized) output and running softmax statistics as each new block of $K,V$ is streamed in. The final result is numerically **identical** to standard attention — this is not an approximation. The speedup comes entirely from cutting the number of slow memory reads/writes, i.e., it's an I/O-aware algorithm, not a lower-FLOP algorithm.

<Callout type="tip" title="The one-sentence idea">
  FlashAttention computes exact attention without ever materializing the full n×n score matrix in HBM — it tiles the computation to fit in fast on-chip SRAM and uses an online (streaming) softmax to combine partial results correctly, trading extra recomputation for far less memory traffic.
</Callout>

## Why Standard Attention Is Memory-Bound

For sequence length $n$ and head dimension $d$, standard attention's HBM traffic (reads + writes of intermediate tensors) is $O(n^2)$ — every element of the $n \times n$ score matrix is written once (after $QK^\top$), read once (for softmax), written once (after softmax), and read again (for the $\times V$ multiply). Compute is also $O(n^2 d)$, but on modern GPUs the ratio of compute throughput to memory bandwidth is high enough that for typical $d$, moving the $n \times n$ matrix dominates wall-clock time — attention is **memory-bandwidth bound**, not compute-bound, for most practical sequence lengths.

## The Online Softmax — The Key Numerical Trick

Softmax over the full row of scores requires knowing the row's maximum (for numerical stability) and the sum of exponentials — both of which normally require seeing the *entire* row before finalizing any output. FlashAttention processes $K, V$ in blocks and updates these statistics incrementally. For a row's scores split into blocks with local max $m^{(1)}, m^{(2)}, \ldots$ and local sums $\ell^{(1)}, \ell^{(2)}, \ldots$, each new block's contribution is folded in by rescaling the running output and running sum by a correction factor $e^{m_{\text{old}} - m_{\text{new}}}$:

$$
m_{\text{new}} = \max(m_{\text{old}}, m_{\text{block}}), \qquad \ell_{\text{new}} = \ell_{\text{old}} \, e^{m_{\text{old}} - m_{\text{new}}} + \ell_{\text{block}} \, e^{m_{\text{block}} - m_{\text{new}}}
$$

$$
O_{\text{new}} = O_{\text{old}} \, e^{m_{\text{old}} - m_{\text{new}}} + \left(P_{\text{block}} V_{\text{block}}\right) e^{m_{\text{block}} - m_{\text{new}}}
$$

This gives a running output that, after the last block, equals *exactly* what a single full-row softmax followed by weighted sum would have produced — no approximation, just algebraic rearrangement so that partial results computed at different times can be correctly combined.

## The Mechanism, Step by Step

1. **Tile** $Q$, $K$, $V$ into blocks small enough that a block of $Q$, plus the current block of $K,V$, plus running statistics all fit in on-chip SRAM together.
2. **Outer loop over $K, V$ blocks** (or $Q$ blocks, depending on kernel variant): load one block of $K,V$ from HBM into SRAM once.
3. **Inner loop over $Q$ blocks**: for each $Q$ block already in SRAM, compute the partial score block $Q_{\text{block}} K_{\text{block}}^\top$, apply the online softmax update above, and accumulate into that $Q$ block's running output — all without ever writing the $n \times n$ score matrix to HBM.
4. **Recompute in the backward pass instead of storing.** Rather than storing the full attention matrix for use in backpropagation (which would defeat the memory savings), FlashAttention recomputes the needed blocks on the fly during the backward pass from the stored softmax statistics — trading a modest amount of extra compute for a large reduction in memory traffic and footprint.
5. **Result:** identical output to standard attention, with HBM traffic reduced from $O(n^2)$ to roughly $O(n^2 d^2 / M)$ where $M$ is SRAM size — in practice a large constant-factor reduction, translating to substantial wall-clock speedups and enabling much longer sequences to fit in memory (since the full score matrix is never materialized).

## Versions: FlashAttention, FlashAttention-2, FlashAttention-3

- **FlashAttention (2022)** introduced the tiling + online softmax approach, framing attention speedups as an I/O-awareness problem rather than a FLOP-reduction problem.
- **FlashAttention-2 (2023)** restructured the parallelism and work partitioning — reducing non-matmul FLOPs, better distributing work across thread blocks and warps (parallelizing over sequence length in addition to batch/heads), and improving GPU occupancy — reported at up to roughly 2× faster than the original FlashAttention.
- **FlashAttention-3 (2024)** targets Hopper-generation GPUs (H100) specifically, exploiting asynchrony between tensor cores and memory copies (warp specialization, overlapping matmul with softmax) and adding low-precision (FP8) support. It reports up to roughly 1.5–2.0× speedup over FlashAttention-2 on H100, reaching up to ~740 TFLOPs/s in FP16 (about 75% of the H100's theoretical peak) and close to 1.2 PFLOPs/s in FP8, with FP8 numerical error reduced substantially versus a naive FP8 baseline through incoherent processing techniques.

<Callout type="note" title="Verified via web search, July 2026">
  FlashAttention-2 is arXiv:2307.08691; FlashAttention-3 is arXiv:2407.08608. Exact speedup figures are hardware- and shape-dependent — treat the numbers above as representative of the papers' reported benchmarks, not universal constants.
</Callout>

## Complexity and Memory Analysis

| Quantity | Standard attention | FlashAttention |
| --- | --- | --- |
| FLOPs | $O(n^2 d)$ | $O(n^2 d)$ — same, no algorithmic FLOP reduction |
| HBM reads/writes | $O(n^2)$ | $O(n^2 d^2 / M)$, i.e., far fewer for realistic SRAM size $M$ |
| Peak activation memory | $O(n^2)$ (materialized score matrix) | $O(n)$ — only running statistics and output kept per block |
| Backward pass | Reuses stored $n \times n$ matrix | Recomputes blocks from saved statistics — extra compute, far less memory |

## Common Mistakes

<ProsCons
  pros={[
    "Produces mathematically exact attention output — not an approximation — so it's a pure speed/memory win with zero accuracy trade-off.",
    "Reduces peak memory from O(n²) to O(n), which is often the deciding factor in how long a context a given GPU can serve, independent of raw speed.",
    "Composable with causal masking, variable-length batching, and multi-query/grouped-query attention — it's an implementation of the same math, not a competing architecture.",
  ]}
  cons={[
    "It optimizes memory *traffic*, not FLOPs — for very short sequences (where the n×n matrix already fits comfortably in SRAM) the benefit shrinks and standard implementations can be competitive.",
    "Kernel implementations are hardware-generation-specific (FlashAttention-3's biggest gains are Hopper-specific); portability to new architectures lags behind release of the hardware.",
    "It does not reduce the underlying O(n²) attention complexity — for extremely long contexts, sub-quadratic architectures or sparse/linear attention variants are still needed on top of it.",
  ]}
/>

- Confusing FlashAttention with a *sparse* or *approximate* attention method — it computes the exact same softmax(QKᵀ/√d)V as vanilla attention; the only change is how it's computed, not what is computed.
- Assuming FlashAttention reduces FLOPs — it reduces memory reads/writes; FLOP count is essentially unchanged (and the backward pass does *more* FLOPs due to recomputation, in exchange for less memory movement).
- Forgetting that its benefits are most visible for longer sequences and larger batches — at very short sequence lengths, the fixed overheads and kernel-launch costs can make the gains marginal.
