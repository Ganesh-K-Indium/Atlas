## Intuition

Self-attention is permutation-invariant by construction, so a Transformer needs *some* explicit signal about token order. The original Transformer added a fixed sinusoidal vector to each token's embedding — position is baked into the *content* of the vector.

RoPE (Rotary Position Embedding) takes a different approach entirely: instead of adding a position vector, it **rotates** the query and key vectors by an angle proportional to their position, before computing attention scores. The elegant payoff: when you dot-product a rotated query against a rotated key, the *absolute* rotation angles cancel out algebraically, leaving only a term that depends on the **relative** distance between the two tokens. Relative position falls out of an absolute-position mechanism for free.

<Callout type="tip" title="The one-sentence idea">
  Rotating both Q and K by their absolute position, then taking their dot product, produces a value that depends only on the difference in positions — because rotation matrices compose: rotating by $m$ then measuring against a rotation by $n$ is equivalent to measuring against a rotation by $n - m$.
</Callout>

## Formal Construction

Split each $d$-dimensional query/key vector into $d/2$ two-dimensional pairs. For pair index $i$, define a rotation frequency:

$$
\theta_i = 10000^{-2i/d}, \qquad i = 0, 1, \ldots, d/2 - 1
$$

Lower-index pairs (small $i$) rotate quickly with position; higher-index pairs rotate slowly — this geometric spread of frequencies (borrowed directly from the original sinusoidal encoding) lets the model represent both fine-grained local position and coarse long-range position.

For a token at position $m$, RoPE applies a block-diagonal rotation matrix $R_{\Theta, m}$ built from $2\times2$ rotation blocks:

$$
R_{\Theta,m}^{(i)} = \begin{pmatrix} \cos(m\theta_i) & -\sin(m\theta_i) \\ \sin(m\theta_i) & \cos(m\theta_i) \end{pmatrix}
$$

applied to each pair of dimensions $(x_{2i}, x_{2i+1})$ of the query and key vectors:

$$
q_m = R_{\Theta,m}\, W_q x_m, \qquad k_n = R_{\Theta,n}\, W_k x_n
$$

## Why the Relative Property Falls Out

Because rotation matrices form a group under composition, $R_{\Theta,m}^\top R_{\Theta,n} = R_{\Theta,n-m}$ — rotating by $m$ and then measuring against a rotation by $n$ is identical to a single rotation by $n - m$. So the attention score:

$$
q_m^\top k_n = (R_{\Theta,m} W_q x_m)^\top (R_{\Theta,n} W_k x_n) = x_m^\top W_q^\top R_{\Theta,n-m} W_k x_n
$$

depends on $m$ and $n$ **only through their difference** $n - m$. This is exactly the relative-position property that earlier approaches (like Transformer-XL's relative position embeddings) had to engineer by hand with extra learned parameters — RoPE gets it as a direct algebraic consequence of using rotations, with zero additional parameters.

## Complex-Number View

The cleanest way to see this: treat each 2D pair $(x_{2i}, x_{2i+1})$ as a single complex number $x_{2i} + i\,x_{2i+1}$. Rotating by angle $m\theta_i$ is just multiplication by $e^{i m \theta_i}$. The query-key dot product for that pair becomes:

$$
\text{Re}\left[ (q_i \, e^{i m \theta_i}) \cdot \overline{(k_i \, e^{i n \theta_i})} \right] = \text{Re}\left[ q_i \bar{k}_i \, e^{i(m-n)\theta_i} \right]
$$

The exponentials combine into $e^{i(m-n)\theta_i}$ — a function of $(m - n)$ alone, confirming the same result via complex arithmetic instead of matrices.

## Minimal PyTorch Implementation

```python
import torch

def rope_frequencies(dim: int, base: float = 10000.0):
    # θ_i = base^(-2i/dim), one frequency per 2D subspace
    i = torch.arange(0, dim, 2).float()
    return 1.0 / (base ** (i / dim))

def apply_rope(x: torch.Tensor, positions: torch.Tensor, freqs: torch.Tensor):
    # x: [seq_len, dim] — pairs (x[..., 0::2], x[..., 1::2]) form each 2D subspace
    angles = positions[:, None] * freqs[None, :]        # [seq_len, dim/2]
    cos, sin = angles.cos(), angles.sin()

    x1, x2 = x[..., 0::2], x[..., 1::2]
    rotated = torch.stack(
        [x1 * cos - x2 * sin, x1 * sin + x2 * cos], dim=-1
    )
    return rotated.flatten(-2)  # interleave back to [seq_len, dim]
```

In practice, `q` and `k` are rotated with this function immediately after the linear projection and before the attention dot product — `v` is left unrotated, since RoPE only needs to affect the *scores*, not the values being aggregated.

RoPE's practical advantage isn't just the elegant math — it composes naturally with **relative** position bias (which is what actually matters for language, where "two words apart" matters more than "at absolute position 47"), while adding zero extra parameters and requiring no changes to the attention computation itself beyond the rotation.
