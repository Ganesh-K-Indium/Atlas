## Intuition

A neural network doesn't consume text — it consumes a sequence of integers, each one an index into a fixed vocabulary. The question a tokenizer answers is: what should the *units* of that vocabulary be?

Whole words are the obvious choice, but language has an unbounded number of them (misspellings, rare names, morphological variants, code identifiers, other languages) — a whole-word vocabulary either explodes in size or drowns in "unknown token" fallbacks. Individual characters or bytes solve the coverage problem completely (there are only so many bytes) but produce very long sequences and force the model to relearn word structure from scratch.

Subword tokenization is the compromise that won: build a vocabulary of frequently occurring *character chunks*, so common words stay whole (one token), rare words decompose into a handful of familiar pieces, and truly novel strings can always fall back to bytes or characters — so nothing is ever literally unrepresentable.

<Callout type="tip" title="The one-sentence idea">
  Start from individual characters (or bytes) as the only guaranteed-covered vocabulary, then greedily merge whichever adjacent pair appears most often in the training corpus — repeating this thousands of times builds a vocabulary that spans from single characters up to whole common words, with everything in between assembled from the merges that mattered most.
</Callout>

## Byte-Pair Encoding (BPE), Derived Step by Step

BPE is an old compression algorithm (Gage, 1994) repurposed for tokenization (Sennrich et al., 2015). The training procedure:

1. **Initialize the vocabulary with the base alphabet.** Every distinct character (or, in the byte-level variant used by GPT-2 and onward, every one of the 256 possible byte values) is a token. Every word in the training corpus starts as a sequence of these base symbols, usually with an end-of-word marker so the tokenizer can tell "est" in "highest" apart from "est" in "establish".
2. **Count all adjacent symbol pairs across the corpus.** For example, if the corpus (as symbol sequences) contains `l o w`, `l o w e r`, `n e w e s t`, `w i d e s t`, count how often each pair like `(l, o)`, `(o, w)`, `(e, s)`, `(s, t)` appears, weighted by word frequency.
3. **Merge the single most frequent pair into a new symbol.** Say `(e, s)` is most frequent — it becomes a new token `es`, and every occurrence of adjacent `e s` in the corpus is replaced by `es`. This merge rule is recorded, in order, in a **merge table**.
4. **Repeat steps 2–3** on the now-updated corpus (which contains the new merged symbols as candidates for further pairing) for a fixed number of merges — this merge count is chosen ahead of time and directly determines the final vocabulary size (base alphabet size + number of merges).
5. **Stop when the merge budget is exhausted.** The final vocabulary is the base alphabet plus every merged symbol created along the way; the ordered merge table itself becomes the tokenizer.

**At inference time**, tokenizing new text means applying the learned merges *in the same order they were learned*: split the input into base characters/bytes, then repeatedly apply the earliest-learned applicable merge until no more merges from the table apply. This is why BPE is deterministic and fast — it's a fixed greedy replay of a lookup table, not a search.

$$
\text{merge}^* = \arg\max_{(a,b)} \; \text{count}(a, b) \quad \text{over all adjacent symbol pairs in the corpus}
$$

## WordPiece: Likelihood Instead of Raw Frequency

WordPiece (used by BERT) follows the same iterative-merge skeleton as BPE but changes the criterion in step 3. Instead of merging whichever pair is simply most *frequent*, WordPiece merges whichever pair most increases the **likelihood of the training corpus** under a unigram language model built from the current vocabulary:

$$
\text{score}(a, b) = \frac{\text{count}(a, b)}{\text{count}(a) \cdot \text{count}(b)}
$$

This scores a candidate merge by how much more often the pair co-occurs than you'd expect if the two symbols were statistically independent — effectively favoring merges that fuse symbols which are unusually "glued together" in the data, rather than merges that are frequent only because their *constituent* symbols happen to be common (like merging two extremely common individual letters into a pair that isn't actually a meaningful unit). In practice this tends to produce slightly different, often more linguistically coherent, subword boundaries than raw-frequency BPE, though the two algorithms are close cousins.

## SentencePiece: Tokenizing the Raw Stream

Both BPE and WordPiece as described above assume a **pre-tokenization** step already happened — the text was already split into words (typically on whitespace) before the merge algorithm ever sees it. That assumption quietly bakes in a specific notion of "word boundary" that doesn't transfer to languages without whitespace-delimited words (Japanese, Chinese, Thai), and it also throws away the information of *where* whitespace occurred, which matters for faithfully reconstructing (detokenizing) the original string.

SentencePiece (Kudo & Richardson, 2018) removes this assumption by treating the input as a **raw, uninterrupted stream of Unicode characters**, whitespace included — it typically encodes the space character itself as an ordinary symbol (visualized as `▁`, U+2581) rather than treating it as a delimiter to strip. BPE or a unigram-language-model algorithm (SentencePiece supports both as the underlying merge/segmentation method) then runs directly on this raw stream. Because whitespace is just another character in the stream, the resulting tokenization is:

- **Language-agnostic** — no assumption that "words" are separated by spaces, so it works uniformly for English, Chinese, and everything in between.
- **Perfectly reversible** — detokenization is just concatenating the token strings and mapping `▁` back to a space, with no heuristics needed to decide where spaces "should" go.

<Callout type="note">
  SentencePiece is a tokenization *framework*, not a competing merge criterion — it can run either the BPE merge algorithm or a probabilistic unigram segmentation algorithm internally. Its distinguishing contribution is the raw-stream input representation, not a new way of scoring merges.
</Callout>

## Code Sketch: Training a Small BPE Tokenizer

```python
from tokenizers import Tokenizer, models, trainers, pre_tokenizers

# Byte-level BPE, the scheme used by GPT-2/GPT-3/GPT-4-style models
tokenizer = Tokenizer(models.BPE())
tokenizer.pre_tokenizer = pre_tokenizers.ByteLevel(add_prefix_space=False)

trainer = trainers.BpeTrainer(
    vocab_size=32000,          # base alphabet (256 bytes) + this many merges
    min_frequency=2,           # ignore pairs seen fewer than 2 times
    special_tokens=["<pad>", "<bos>", "<eos>", "<unk>"],
)

files = ["corpus.txt"]
tokenizer.train(files, trainer)

encoded = tokenizer.encode("The tokenizer's merges determine everything downstream.")
print(encoded.tokens)
# e.g. ['The', 'Ġtoken', 'izer', "'s", 'Ġmerges', 'Ġdetermine', ...]
```

The `Ġ` (or `▁` in SentencePiece) marks a token that begins a new word, preserving the whitespace information that a naive character split would lose — this is exactly the detail that lets detokenization reconstruct the original string exactly.

  pros={[
    "Subword tokenization guarantees full coverage — any string can be represented, down to individual bytes if nothing else matches.",
