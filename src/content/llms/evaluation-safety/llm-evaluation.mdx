## Intuition

A benchmark score is a proxy, not the thing itself. When a lab reports "our model scores 88% on MMLU," that number is standing in for a much fuzzier target — "how good is this model at knowledge and reasoning tasks in general." The gap between the proxy and the target is where almost every evaluation controversy in LLM research lives: a model can score well on the proxy by genuinely being good at the target, or by having memorized the proxy's test questions, or by being specifically tuned to the proxy's exact format, none of which are distinguishable from the score alone.

<Callout type="tip" title="The one-sentence version">
  Two entirely different things get called "LLM evaluation": scoring a model against a fixed, gradeable answer key (static benchmarks), and asking a strong model to judge another model's open-ended, ungradeable output against a rubric (LLM-as-a-judge). They fail in different ways, and a rigorous evaluation setup usually needs both.
</Callout>

## Two Fundamentally Different Evaluation Paradigms

### Static benchmark evaluation

Static benchmarks give the model a fixed set of questions with a fixed, objectively checkable answer — multiple choice, exact-match short answer, or unit-test-verified code. The scoring function is deterministic: given the model's output and the reference answer, you compute a 0/1 (or partial-credit) score with no further judgment call required.

The canonical example is **MMLU** (Massive Multitask Language Understanding, Hendrycks et al. 2020) — roughly 14,000 multiple-choice questions spanning 57 subjects from elementary math to professional law and medicine. Its appeal is exactly its rigidity: the grading is unambiguous, reproducible across labs, and cheap to run at scale. The same logic underlies exact-match QA benchmarks and unit-test-based code benchmarks (e.g. HumanEval-style pass@k scoring).

The weakness is equally structural. Multiple-choice format rewards a specific skill (picking the best of four options) that doesn't transfer cleanly to open-ended generation, real questions rarely arrive as four-way multiple choice, and a fixed, public answer key is precisely the kind of artifact that can leak into training data (more on this below).

### LLM-as-a-judge evaluation

Most of what people actually want from a production LLM — a helpful, well-reasoned, appropriately-toned response to an open-ended prompt — has no single correct answer to check against. **LLM-as-a-judge** evaluation addresses this by using a strong "judge" model to score a candidate model's free-form output against a rubric, often by comparing two candidate responses head-to-head and asking the judge to pick a winner (or score each independently on a 1–10 scale).

Zheng et al. (2023), in the paper that introduced **MT-Bench** and analyzed **Chatbot Arena**, showed that a strong judge model (GPT-4, at the time) agreed with human preference judgments over 80% of the time — comparable to the agreement rate between two different human raters. That result is why LLM-as-a-judge became a standard, not niche, evaluation technique: it scales far better than paying humans to read every response, while correlating well enough with human judgment to be a useful signal.

The same paper is equally important for cataloging the technique's biases:

- **Position bias** — judges tend to favor whichever response is shown first (or, in some setups, last) independent of quality, which is why serious harnesses randomize or swap response order and average.
- **Verbosity bias** — judges tend to rate longer, more elaborate responses more favorably even when the extra length adds no real information.
- **Self-enhancement bias** — a judge model tends to rate outputs written in its own "style" (or by itself) more favorably than outputs from other model families.
- **Limited reasoning ability** — a judge is not immune to the same reasoning failures as the model it's judging, particularly on math and multi-step logic, so a judge's confident-sounding verdict on a reasoning-heavy answer deserves real skepticism.

<Callout type="warning" title="A judge is a model, not an oracle">
  LLM-as-a-judge is a genuinely useful scaling technique, not a source of ground truth. Any pipeline that reports "judge win rate" numbers should also report the judge model used, whether position was randomized, and ideally a spot-check correlation against human ratings — otherwise the number is only as trustworthy as the judge's own blind spots.
</Callout>

## The Contamination and Saturation Problem

Static benchmarks have a second-order failure mode that's independent of format: **benchmark contamination**, where a model's training data accidentally (or, in some allegations, not-so-accidentally) contains the benchmark's test questions and answers. A model that has seen a question during pretraining can answer it correctly through memorization, not the underlying capability the benchmark was designed to measure — and the resulting score tells you almost nothing about how the model performs on questions it hasn't seen.

This isn't a hypothetical concern. Studies auditing MMLU specifically have found meaningful contamination — one widely cited analysis estimated on the order of a quarter to a third of MMLU questions show detectable overlap with common pretraining corpora, and re-running evaluation on a decontaminated "clean" mirror of GSM8K-style math questions has been shown to drop some models' scores by double-digit percentage points. Public accusations of contamination (for instance, community allegations around specific frontier releases having been tuned toward benchmark-specific data) have become recurring events in the field, even when the accused lab disputes the claim — the point isn't any single incident, it's that the underlying risk is structural: any benchmark whose questions and answers are published on the open web can, in principle, end up in a future pretraining crawl.

A related but distinct problem is **benchmark saturation**: even without contamination, once most frontier models cluster near the ceiling of a benchmark (95%+ on MMLU, for example), the benchmark stops discriminating between models of meaningfully different capability — everyone is bunching up near 100%, and the benchmark's remaining few points of "headroom" say little.

$$
\text{Reported score} = \underbrace{\text{true capability}}_{\text{what you want to measure}} + \underbrace{\text{memorization effect}}_{\text{contamination}} + \underbrace{\text{format-specific tuning}}_{\text{overfitting to benchmark style}}
$$

The reported number conflates all three terms, and only the first one is what most people think they're reading off the leaderboard.

### Mitigation: held-out, private, and continuously-refreshed evaluation

The evaluation community's dominant response has been to stop relying solely on static, permanently-public test sets and instead use benchmarks that are refreshed or gated in ways that make contamination structurally harder:

- **Continuously-refreshed benchmarks** such as **LiveBench** release new questions on a rolling basis (drawn from recent arXiv papers, news, and datasets published after any given model's training cutoff), and score with objective, automatically-checkable ground truth rather than an LLM judge — so a model can't have memorized a question that didn't exist yet when it was trained.
- **Temporal-filtering benchmarks** such as **LiveCodeBench** collect coding problems from live competitive-programming sites (LeetCode, AtCoder, Codeforces) tagged by release date, and evaluate a model only on problems released after its training cutoff — turning contamination-checking into a built-in property of the evaluation design rather than an after-the-fact audit.
- **Private or held-out test sets**, where the questions are never published at all (only aggregate scores are released, sometimes by a third party running the eval on a lab's behalf), remove the leak vector entirely at the cost of external reproducibility and auditability.

None of these fully "solve" evaluation — a continuously-refreshed benchmark still has to be well-designed, and a private test set requires trusting whoever controls it — but they directly target the specific failure mode of "the answer key is public and old enough to have been crawled."

<Callout type="note" title="See also">
  The [Frontier Model Landscape](/docs/llms/frontier-landscape/frontier-model-landscape) page discusses this same caution from the buyer's side: don't trust a single leaderboard snapshot, and cross-check any headline benchmark claim against independent evaluations before it informs a model choice.
</Callout>

## A Minimal LLM-as-a-Judge Harness

The core pattern — bias-aware, rubric-based, order-randomized — is simple enough to sketch directly:

```python
import random

JUDGE_PROMPT = """You are an impartial judge evaluating an AI assistant's response.

[User Question]
{question}

[Assistant Response]
{response}

[Rubric]
Score the response from 1-10 on each of: correctness, completeness, clarity.
Then give an overall score from 1-10. Be skeptical of confident-sounding but
factually wrong reasoning. Do not reward length for its own sake.

Respond in JSON: {{"correctness": _, "completeness": _, "clarity": _, "overall": _, "rationale": "_"}}
"""

def judge_pairwise(question, response_a, response_b, judge_model_call):
    """Head-to-head comparison with order randomization to mitigate position bias."""
    swapped = random.random() < 0.5
    first, second = (response_b, response_a) if swapped else (response_a, response_b)

    prompt = f"""Question: {question}

Response 1: {first}
Response 2: {second}

Which response is better? Answer "1", "2", or "tie", then explain briefly."""

    verdict = judge_model_call(prompt)
    # Un-swap the verdict before returning, so callers always get results
    # in terms of the original (a, b) ordering — never trust a single,
    # non-randomized ordering's verdict at face value.
    return unswap(verdict, swapped)
```

The details that matter more than the prompt wording: randomizing (and un-swapping) response order every single call, using a judge model from a *different* family than any of the candidates being judged where possible (to reduce self-enhancement bias), and treating the judge's numeric score as a noisy signal to average over many samples rather than a precise measurement from one call.
