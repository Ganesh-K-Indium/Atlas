## Intuition

A base model fresh out of pretraining is a very good next-token predictor over internet-scale text — but "predict what comes next" and "answer this question helpfully" are not the same task. Asked a question, a base model is just as likely to continue it with more questions, a Wikipedia-style continuation, or a shift in topic, because that's what often follows similar text in its training corpus.

Supervised Fine-Tuning (SFT) closes that gap without inventing a new objective. It reuses exactly the same next-token cross-entropy loss from pretraining, but changes *what data* it's applied to: instead of raw internet text, the model trains on a curated set of (prompt, response) pairs written or selected specifically to demonstrate the desired instruction-following behavior. The mechanism doesn't change; the data distribution does.

<Callout type="tip" title="The one-sentence idea">
  SFT is pretraining's next-token objective continued on a small, curated dataset of instruction-response pairs, with the loss masked so gradients only flow from predicting the response tokens — the prompt is context to condition on, not something the model needs to learn to generate.
</Callout>

## The SFT Objective: Loss Masking

Given a formatted example consisting of a prompt $x$ and a target response $y = (y_1, \ldots, y_m)$, SFT still trains with autoregressive cross-entropy — but only over the response tokens:

$$
\mathcal{L}_{\text{SFT}}(\theta) = -\sum_{t=1}^{m} \log p_\theta(y_t \mid x, y_1, \ldots, y_{t-1})
$$

Note what's absent from this sum: there's no term for predicting the tokens of the prompt $x$ itself. In implementation, this is done by constructing a **loss mask** the same length as the full tokenized sequence (prompt + response), set to 0 over prompt positions and 1 over response positions, and multiplying the per-token loss by this mask before averaging:

$$
\mathcal{L}_{\text{SFT}}(\theta) = -\frac{1}{\sum_t m_t}\sum_{t=1}^{T} m_t \cdot \log p_\theta(x_t \mid x_{<t}), \qquad m_t = \begin{cases} 0 & t \text{ in prompt} \\ 1 & t \text{ in response} \end{cases}
$$

The prompt tokens are still fed through the model as context (the model attends to them normally via causal self-attention) — they're excluded only from the *loss*, not from the forward pass. This is the detail that's easy to get wrong: masking a token out of attention would remove it as context; masking it out of the loss just means the model isn't penalized for how well it "predicts" text it was only ever supposed to condition on.

## Why Mask the Prompt at All?

Without masking, the model would also be trained to predict the prompt tokens from the tokens before them — but prompts in an instruction dataset are usually written by humans and vary enormously in phrasing; there's no benefit to the model learning to "predict" a user's question verbatim, and doing so wastes capacity and gradient signal on a task (generating plausible-looking prompts) that has nothing to do with the actual goal (generating good responses). Masking focuses every unit of gradient signal on the part of the sequence the model will actually need to generate at inference time.

## Data: What Makes a Good SFT Dataset

SFT datasets are comparatively small relative to pretraining corpora — typically thousands to low tens-of-thousands of examples rather than trillions of tokens — but their quality matters disproportionately, since every single example directly shapes the model's response style and behavior. Sources typically include:

- **Human-written demonstrations**: annotators write example responses to a diverse set of prompts, following detailed labeling guidelines about tone, format, and refusal behavior.
- **Filtered/curated existing data**: high-quality question-answer or instruction-following text mined from existing corpora and filtered for quality.
- **Model-generated data with human or model-based filtering**: increasingly common — using a strong existing model to draft candidate responses, then filtering or lightly editing them, which is far cheaper than pure human authorship at scale.

<Callout type="note">
  SFT is a necessary but not sufficient step for building a well-aligned assistant. It teaches format and surface-level instruction-following well, but SFT alone tends to underperform on subtler preferences (which of two plausible-sounding responses is actually better, how to handle ambiguous or borderline requests) — that's the gap RLHF and DPO are built to close on top of an SFT model.
</Callout>

## Code Sketch: Loss Masking in Practice

```python
import torch

def build_labels(input_ids: torch.Tensor, prompt_lengths: torch.Tensor, pad_token_id: int):
    """
    input_ids: [batch, seq_len] — tokenized (prompt + response), padded
    prompt_lengths: [batch] — number of prompt tokens per example
    Returns labels with prompt tokens and padding set to -100 (ignored by
    PyTorch's cross-entropy loss).
    """
    labels = input_ids.clone()
    for i, plen in enumerate(prompt_lengths):
        labels[i, :plen] = -100          # mask the prompt out of the loss
    labels[input_ids == pad_token_id] = -100  # mask padding too
    return labels

# Standard HF training loop then just does:
# outputs = model(input_ids=input_ids, attention_mask=attention_mask, labels=labels)
# loss = outputs.loss   # cross-entropy, automatically ignoring -100 positions
```

`-100` is PyTorch's (and Hugging Face's) convention for "ignore this position in the loss" — it's the mechanism that implements the mask described above without needing a custom loss function.

## Common Mistakes

  pros={[
    "Reuses the pretraining infrastructure and objective almost unchanged — no new loss function or training loop architecture needed.",
    "Directly teaches format, tone, and instruction-following conventions from a relatively small, controllable dataset.",
    "A necessary, well-understood, comparatively cheap step relative to RLHF/DPO's preference-data collection and (in RLHF's case) RL infrastructure.",

  cons={[

    "Treating SFT as the final alignment step rather than a foundation — SFT alone doesn't optimize for the fact that some correct-format responses are still meaningfully better than other correct-format responses, which is exactly the gap preference-based methods address.",
