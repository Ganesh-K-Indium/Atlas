## Intuition

Full fine-tuning updates every parameter in a pretrained model — for a modern LLM, that's billions of weights, each needing its own optimizer state (Adam alone stores two extra values per parameter), which is why full fine-tuning a 70B model needs hundreds of gigabytes of GPU memory.

**LoRA (Low-Rank Adaptation)** starts from an empirical observation: the *update* a model needs during task-specific fine-tuning tends to have low "intrinsic rank" — the change in behavior can be captured by a much lower-dimensional transformation than the full weight matrix. So instead of learning a full-rank update $\Delta W$, LoRA learns a **low-rank decomposition** of it, and freezes everything else.

<Callout type="tip" title="The one-sentence idea">
  Freeze the pretrained weight matrix $W$ entirely, and learn only a low-rank correction $\Delta W = BA$ where $B$ and $A$ are much smaller matrices — reducing trainable parameters by orders of magnitude while leaving the original model's knowledge untouched.
</Callout>

## Formal Construction

For a pretrained weight matrix $W_0 \in \mathbb{R}^{d \times k}$, LoRA reparameterizes the fine-tuned weight as:

$$
W = W_0 + \Delta W = W_0 + BA
$$

where $B \in \mathbb{R}^{d \times r}$, $A \in \mathbb{R}^{r \times k}$, and the rank $r \ll \min(d, k)$ — commonly $r \in \{4, 8, 16, 64\}$ versus $d, k$ in the thousands. $W_0$ is **frozen** (no gradient computed, no optimizer state stored); only $A$ and $B$ are trained.

The forward pass becomes:

$$
h = W_0 x + \Delta W x = W_0 x + B(Ax) \cdot \frac{\alpha}{r}
$$

where $\alpha$ is a scaling hyperparameter (analogous to a learning-rate multiplier for the adapter) that keeps the update's magnitude comparable across different choices of $r$. $A$ is typically initialized with small random values (Gaussian) and $B$ with **zeros**, so that $\Delta W = 0$ at the start of training — the model behaves identically to the pretrained checkpoint before any adapter training happens, guaranteeing a stable starting point.

**Why this reduces trainable parameters so drastically:** for a $4096 \times 4096$ attention projection matrix ($d = k = 4096$), full fine-tuning trains $16.8$M parameters. LoRA with $r=8$ trains $A$ ($8 \times 4096$) and $B$ ($4096 \times 8$) — about $65$K parameters, a **~250× reduction** for that single matrix.

## Which Weights to Adapt

The original paper found that applying LoRA to just the **attention projection matrices** ($W_q, W_k, W_v, W_o$) — leaving the feed-forward layers frozen entirely — captures most of the benefit at a fraction of the parameter cost. Later work generally extends LoRA to feed-forward layers too when maximum quality is needed, trading a modest parameter increase for better task adaptation.

## QLoRA: Quantize the Frozen Weights Too

LoRA already avoids storing optimizer state for the frozen weights, but the frozen weights themselves still sit in GPU memory at full precision (16-bit). **QLoRA** (Dettmers et al., 2023) goes further: it quantizes the frozen base model to **4-bit precision**, then trains LoRA adapters in 16-bit on top — backpropagating gradients *through* the frozen 4-bit weights into the full-precision adapters, without ever fully dequantizing the base model into memory.

QLoRA introduces three specific innovations:

1. **4-bit NormalFloat (NF4):** a quantization data type designed for weights that follow a (roughly) zero-centered normal distribution — the typical shape of pretrained neural network weights. Instead of uniform quantization bins, NF4 places quantization levels at the quantiles of a standard normal distribution, which is information-theoretically optimal for normally-distributed inputs and measurably more accurate than plain 4-bit integer quantization at the same bit width.
2. **Double quantization:** the quantization *constants themselves* (one scale factor per block of weights) are quantized again, saving roughly an additional 0.4 bits per parameter on average — a small-sounding number that adds up across billions of parameters.
3. **Paged optimizers:** uses NVIDIA unified memory to automatically page optimizer states to CPU RAM when GPU memory spikes (e.g., from long-sequence gradient checkpointing), preventing out-of-memory crashes rather than reducing average memory use.

## Minimal Implementation Sketch

```python
import torch
import torch.nn as nn

class LoRALinear(nn.Module):
    def __init__(self, base: nn.Linear, r: int = 8, alpha: int = 16):
        super().__init__()
        self.base = base
        for p in self.base.parameters():
            p.requires_grad = False          # freeze the pretrained weight

        d_out, d_in = base.weight.shape
        self.A = nn.Parameter(torch.randn(r, d_in) * 0.01)
        self.B = nn.Parameter(torch.zeros(d_out, r))  # zero init -> ΔW = 0 initially
        self.scale = alpha / r

    def forward(self, x):
        frozen_out = self.base(x)
        lora_out = (x @ self.A.T) @ self.B.T
        return frozen_out + self.scale * lora_out
```

In production, libraries like Hugging Face `peft` and `bitsandbytes` handle the 4-bit quantization, adapter injection, and merging automatically — but this sketch shows the entire mechanism: freeze, decompose, zero-init, add.

    "Dramatically lower memory and storage cost — adapters for a 7B model are typically tens of megabytes, versus tens of gigabytes for a full fine-tuned checkpoint.",
    "No inference latency penalty if the adapter is merged back into W₀ post-training (W = W₀ + BA is just a matrix add, done once).",
    "Enables serving many task-specific adapters against one shared frozen base model, which is far cheaper than hosting many fully fine-tuned models.",
