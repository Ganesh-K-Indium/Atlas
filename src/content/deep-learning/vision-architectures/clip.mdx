## Intuition

Standard image classifiers are trained to predict one of a fixed, predetermined set of class labels — decided in advance, baked into the final classification layer's shape. Add a new class and you must collect labeled examples and retrain (or at least fine-tune) that layer. CLIP's premise: instead of training to predict *labels*, train a model to predict whether an image and a piece of *natural-language text* describe the same thing. Do that well enough, over a large and diverse enough set of (image, text) pairs scraped from the internet, and the model never needs a fixed label set at all — any class can be expressed as a text description at inference time, with no retraining.

<Callout type="tip" title="The one-sentence idea">
  Train an image encoder and a text encoder jointly so that the embeddings of a matching image-caption pair are pulled close together and embeddings of mismatched pairs are pushed apart — then classify a new image, zero-shot, by seeing which class-name text embedding it's closest to, with no classification layer ever trained for those specific classes.
</Callout>

## The Contrastive Pretraining Objective

CLIP trains two separate encoders in parallel: an image encoder $f$ (a CNN like ResNet, or a ViT) producing image embeddings, and a text encoder $g$ (a Transformer) producing text embeddings, both projected into a shared $d$-dimensional embedding space. Given a batch of $N$ (image, text) pairs scraped from the web — a caption naturally paired with its image — CLIP computes image embeddings $\{v_1, \ldots, v_N\}$ and text embeddings $\{u_1, \ldots, u_N\}$ (both L2-normalized), then the $N \times N$ matrix of cosine similarities:

$$
\text{sim}(v_i, u_j) = \frac{v_i \cdot u_j}{\|v_i\| \|u_j\|}
$$

Only the $N$ diagonal entries $(v_i, u_i)$ are true matching pairs; all $N^2 - N$ off-diagonal entries are mismatched (image $i$ paired with some other caption $j \neq i$) — and crucially, CLIP doesn't need these negatives labeled or curated: *any* other caption in the same batch is a valid negative, since it's very unlikely to describe image $i$. The loss treats this as a symmetric classification problem over the batch, applying InfoNCE-style cross-entropy in both directions:

$$
\mathcal{L} = \frac{1}{2N} \sum_{i=1}^{N} \left[ -\log \frac{\exp(\text{sim}(v_i, u_i) / \tau)}{\sum_{j=1}^{N} \exp(\text{sim}(v_i, u_j) / \tau)} - \log \frac{\exp(\text{sim}(v_i, u_i) / \tau)}{\sum_{j=1}^{N} \exp(\text{sim}(u_i, v_j) / \tau)} \right]
$$

The first term is the standard cross-entropy loss of a classifier that, given image $i$'s embedding, must pick out its true caption $u_i$ from among all $N$ captions in the batch (image-to-text direction); the second term is the same thing in reverse — given caption $i$, pick out its true image from among all $N$ images (text-to-image direction). $\tau$ is a learned temperature that controls how peaked the softmax distribution is. Averaging both directions (hence "symmetric") ensures the objective pulls matched pairs together and pushes apart mismatched pairs from *both* modalities' perspectives, not just one.

Why this works with noisy, uncurated web data: the objective never requires a fixed taxonomy of classes or manually verified negative pairs — it only needs "this caption was found next to this image," and treats every other pair in a large batch as an (imperfect but statistically good enough) negative. Larger batch sizes give harder, more numerous negatives and are one of the main levers CLIP-style training uses to improve embedding quality.

## Zero-Shot Classification — Why the Contrastive Objective Enables It

Because CLIP's image and text encoders are trained to place semantically matching content close together in one *shared* embedding space — regardless of which specific classes appeared during pretraining — a new classification task can be solved without training a single new weight:

1. Take the target class names (e.g. "cat", "dog", "airplane") and turn each into a short natural-language prompt, e.g. `"a photo of a {class}."` (prompt engineering matters here — the exact phrasing measurably affects accuracy, since the text encoder was trained on caption-like sentences, not bare class names).
2. Encode every prompt with the text encoder to get one embedding per candidate class.
3. Encode the query image with the image encoder to get its embedding.
4. Compute cosine similarity between the image embedding and every class-prompt embedding; predict the class whose prompt embedding is most similar.

This works precisely *because* the pretraining objective never tied the embedding space to a fixed label set — it only ever learned "these two things co-occurred, these did not," which generalizes to any new text description at inference time, including class names never seen paired with any image during training. A conventional classifier's final layer, by contrast, has a fixed number of output units baked in at training time and has no mechanism at all for handling a class it wasn't trained on.

## Code Sketch

```python
import torch
import torch.nn.functional as F

def clip_loss(image_embeds, text_embeds, temperature):
    """image_embeds, text_embeds: (N, d), L2-normalized. Matched pairs share index i."""
    logits = image_embeds @ text_embeds.T / temperature   # (N, N) similarity matrix
    labels = torch.arange(logits.size(0), device=logits.device)  # diagonal = matches

    loss_i2t = F.cross_entropy(logits, labels)      # image -> correct text
    loss_t2i = F.cross_entropy(logits.T, labels)    # text -> correct image
    return (loss_i2t + loss_t2i) / 2

def zero_shot_classify(image, class_names, image_encoder, text_encoder, tokenizer):
    prompts = [f"a photo of a {name}." for name in class_names]
    text_embeds = F.normalize(text_encoder(tokenizer(prompts)), dim=-1)     # (C, d)
    image_embed = F.normalize(image_encoder(image.unsqueeze(0)), dim=-1)   # (1, d)
    similarities = image_embed @ text_embeds.T                              # (1, C)
    return similarities.argmax(dim=-1)   # predicted class index
```
