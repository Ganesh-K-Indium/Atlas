## Intuition

A standard (dense) Transformer feed-forward block applies the *same* set of weights to every token — every token's forward pass touches every parameter in that block. Total parameter count and per-token compute cost are therefore locked together: to add more capacity (parameters), you necessarily add more compute per token.

**Mixture of Experts (MoE)** breaks this coupling. Instead of one feed-forward block, an MoE layer has many parallel feed-forward blocks ("experts"), and a small learned **gating network** decides, per token, which few experts should process that specific token. Only the selected experts' parameters are used in that token's forward pass — the rest sit idle for that token. Total parameter count can now grow by adding more experts, while per-token compute stays roughly fixed (bounded by however many experts are selected per token, not the total number available).

<Callout type="tip" title="The one-sentence idea">
  Replace one dense feed-forward block with many expert feed-forward blocks plus a gating network that routes each token to only a small subset (top-k) of them — so total parameters scale with the number of experts, while compute per token scales only with k, decoupling model capacity from inference cost.
</Callout>

## Deriving Sparse Top-k Routing

Let there be $N$ expert feed-forward networks $E_1, \dots, E_N$, each an ordinary feed-forward block (same shape as a dense Transformer's FFN). For an input token representation $x$, a **gating network** — typically just a learned linear projection followed by softmax — produces a score for every expert:

$$
g(x) = \text{softmax}\big(W_g x\big) \in \mathbb{R}^N
$$

A *dense* mixture (using every expert, weighted by $g$) would defeat the purpose — it still touches every expert's parameters per token. The sparse version keeps only the **top-$k$** entries of $g(x)$ (commonly $k=1$ or $k=2$) and zeroes out the rest before renormalizing, so the token's output is a weighted combination of only the selected experts:

$$
\mathcal{G}(x) = \text{softmax}\big(\text{TopK}(W_g x,\ k)\big), \qquad y = \sum_{i=1}^{N} \mathcal{G}(x)_i \cdot E_i(x)
$$

where $\text{TopK}(v, k)$ keeps the $k$ largest entries of $v$ and sets the rest to $-\infty$ (so they vanish under softmax). Crucially, **$E_i(x)$ is only computed for the experts with nonzero gate weight** — the sum's other terms are exactly zero and their corresponding expert networks are never invoked for this token, not merely down-weighted. This is what makes the routing genuinely sparse rather than just a soft-weighted dense mixture.

## Why This Decouples Parameter Count from Per-Token Compute

Consider a layer with $N$ experts, each with $P$ parameters, using top-$k$ routing:

- **Total parameters** in the layer: $N \times P$ (every expert's weights must be stored, since any token might route to any expert across a batch).
- **Compute (FLOPs) per token**: $k \times P$ — only the $k$ selected experts actually run a forward pass for that token, regardless of how large $N$ is.

So increasing $N$ (adding more experts) grows the model's total parameter count and its representational capacity — different experts can specialize on different kinds of tokens or sub-tasks — **without** increasing the compute cost of processing any individual token, since $k$ stays fixed. This is precisely the property that lets frontier MoE-based LLMs report total parameter counts far larger than the number of parameters actually active per forward pass — see [Frontier Model Landscape](/docs/llms/frontier-landscape/frontier-model-landscape) for how this shows up in reported total-vs-active parameter counts for real deployed models. A dense model with the same *total* parameter count would require that entire count in FLOPs per token; an MoE model with the same total count only pays for $k/N$ of it.

## The Load-Balancing Problem

Naive top-k routing, trained with only the task loss, has a well-documented failure mode: the gating network can converge to routing most or all tokens to a small handful of "favorite" experts, while the remaining experts receive few or no tokens and never receive enough gradient signal to become useful. This happens because of a **rich-get-richer dynamic**: an expert that happens to perform slightly better early in training (by chance of initialization) gets selected more often, which gives it more gradient updates, which makes it perform even better, which makes the gate select it even more — a positive feedback loop with nothing in the plain task loss to counteract it. The practical consequences are twofold: most of the model's added parameters go unused (defeating the purpose of adding experts), and the few overloaded experts become a compute/memory bottleneck in distributed training and serving, where experts are typically sharded across devices and an imbalanced load means some devices sit idle while others are overloaded.

**The auxiliary load-balancing loss.** The standard fix (introduced in Shazeer et al. 2017's sparsely-gated MoE and used in essentially all subsequent MoE training recipes) adds an auxiliary loss term that explicitly penalizes uneven routing across a batch. A common formulation defines, per expert $i$: $f_i$, the fraction of tokens in the batch routed to expert $i$, and $P_i$, the average gate probability mass assigned to expert $i$ across the batch (a differentiable proxy, since $f_i$ itself — a hard count — has no useful gradient). The auxiliary loss is proportional to:

$$
L_{\text{aux}} = N \sum_{i=1}^{N} f_i \cdot P_i
$$

This is minimized (for a fixed total token count) when every expert receives an equal share of tokens ($f_i = 1/N$ for all $i$) — a balanced load is the loss's optimum, so adding this term as an auxiliary objective (weighted by a small coefficient and summed with the main task loss) actively discourages the collapse-to-a-few-experts dynamic, without requiring any hard routing constraint that would otherwise break differentiability.

## Implementation Sketch

```python
import torch
import torch.nn as nn
import torch.nn.functional as F

class MoELayer(nn.Module):
    def __init__(self, d_model, d_ff, num_experts, top_k=2):
        super().__init__()
        self.experts = nn.ModuleList([
            nn.Sequential(nn.Linear(d_model, d_ff), nn.ReLU(), nn.Linear(d_ff, d_model))
            for _ in range(num_experts)
        ])
        self.gate = nn.Linear(d_model, num_experts)
        self.num_experts = num_experts
        self.top_k = top_k

    def forward(self, x):
        # x: [batch * seq_len, d_model] -- flattened tokens
        logits = self.gate(x)                              # [tokens, num_experts]
        topk_vals, topk_idx = logits.topk(self.top_k, dim=-1)
        topk_weights = F.softmax(topk_vals, dim=-1)          # renormalize over the k selected

        # load-balancing auxiliary loss
        gate_probs = F.softmax(logits, dim=-1)               # P_i proxy: mean gate prob per expert
        chosen_mask = F.one_hot(topk_idx, self.num_experts).sum(dim=1).float()  # [tokens, num_experts]
        f_i = chosen_mask.mean(dim=0)                         # fraction of tokens routed to each expert
        P_i = gate_probs.mean(dim=0)
        aux_loss = self.num_experts * (f_i * P_i).sum()

        out = torch.zeros_like(x)
        for slot in range(self.top_k):
            expert_idx = topk_idx[:, slot]                    # which expert each token picked in this slot
            weight = topk_weights[:, slot].unsqueeze(-1)
            for e in range(self.num_experts):
                mask = expert_idx == e
                if mask.any():
                    out[mask] += weight[mask] * self.experts[e](x[mask])  # only run expert e on ITS tokens
        return out, aux_loss
```

The inner loop only calls `self.experts[e]` on the subset of tokens actually routed to it — in a real distributed implementation, this corresponds to tokens being physically dispatched to whichever device holds expert $e$'s weights, which is the practical mechanism behind the "only active experts run" property.

## Common Mistakes

<ProsCons
  pros={[
    "Massively increases model capacity (total parameters) without a proportional increase in per-token inference/training compute.",
    "Experts can specialize -- different experts empirically tend to handle different kinds of tokens or sub-tasks, though the specific specialization pattern is not always interpretable or guaranteed.",
    "Enables scaling total model size largely independently of the compute budget available for a single forward pass.",
  ]}
  cons={[
    "Total memory footprint (all experts must be stored, e.g. across devices) can be very large even though compute per token is small -- MoE trades compute for memory/storage, not memory.",
    "Requires the auxiliary load-balancing loss (and careful tuning of its weight) to avoid routing collapse -- without it, most of the added parameters go unused.",
    "Distributed serving/training requires routing tokens across devices holding different experts, adding real systems complexity (communication overhead) not present in dense models.",
    "Routing decisions are discrete (top-k), which is inherently non-smooth -- gradients flow through the gate's softmax weights but not through the hard selection itself, an approximation that can create training instabilities specific to MoE.",
  ]}
/>

A common mistake is assuming an MoE model's total parameter count is a fair comparison point against a dense model's parameter count for compute purposes — the relevant number for inference cost is the *active* parameter count ($k/N$ fraction of the total), not the total.
