## Intuition

Every major cloud provider has converged on offering both "train your own model" infrastructure and "call a hosted foundation model" access — but they've drawn the boundary between the two in different places, and that boundary shapes how you architect around each platform. [AWS for ML](/docs/mlops/cloud-big-data/aws-for-ml) draws it as two separately branded products, SageMaker and Bedrock. Azure and Google Cloud draw it differently, and each choice has real consequences for how a team structures its ML organization and toolchain.

<Callout type="tip" title="The one-sentence idea">
  Azure separates *application building on top of foundation models* (Azure AI Foundry) from *training custom models from scratch* (Azure Machine Learning) as two distinct platforms with different target users; Google Cloud instead unifies both under one platform (Vertex AI, now being folded into the broader "Gemini Enterprise" branding) on the argument that owning the full stack — custom silicon, frontier models, and cloud infrastructure — makes a single integrated surface the right design.
</Callout>

## Azure: Two Platforms for Two Different Jobs

**Azure AI Foundry** consolidates what used to be separate Azure AI Studio and Azure OpenAI Studio experiences into one workspace for building AI applications and agents — chatbots, document-processing pipelines, RAG assistants — primarily on top of pre-trained or fine-tuned large language models accessed through a shared model catalog spanning OpenAI, Meta, Mistral, and other providers.

**Azure Machine Learning (Studio)** remains the platform for the classic ML lifecycle: training predictive models on tabular data, custom computer vision, time-series forecasting, and any workflow where the model is trained from scratch on your own architecture rather than adapted from someone else's foundation model. It provides managed compute clusters, experiment tracking, AutoML, pipelines, and managed online/batch endpoints — conceptually parallel to SageMaker's role on AWS.

The architectural implication: on Azure, "which product do I use" maps directly onto the same build-vs-consume question as AWS's SageMaker/Bedrock split, just with different names and a slightly different center of gravity — Foundry leans further into agent orchestration and application-layer concerns (evaluation, tracing, governance for LLM apps) than Bedrock's narrower model-invocation focus.

## Google Cloud: One Platform, Vertically Integrated

<Mermaid chart={`
graph TD
  Garden["Model Garden\n(200+ models: Gemini, Gemma, partner + OSS models)"] --> Choice{"Use as-is via API,\nor customize?"}
  Choice -->|"Use as-is"| Endpoint1["Managed endpoint"]
  Choice -->|"Customize"| Tune["Fine-tuning / custom training"]
  Tune --> Registry["Vertex AI Model Registry"]
  Registry --> Endpoint2["Vertex AI Endpoint"]
  Data["Your training data"] --> Custom["Vertex AI Custom Training"]
  Custom --> Registry
  Endpoint1 --> Agents["Agent Builder / orchestration"]
  Endpoint2 --> Agents
`} />

**Vertex AI** has historically been Google's single unified platform spanning both ends of the spectrum in one product: **Model Garden** gives access to 200+ models (Google's own Gemini and Gemma families alongside third-party and open-source models, including Anthropic's Claude) for teams that just want to call a hosted model, while **Vertex AI Custom Training** and the **Model Registry** give full control to teams training bespoke models on their own data and deploying them to Vertex AI Endpoints — both paths converge on the same registry, endpoint, and pipeline tooling rather than living in separately branded products.

Google's strategic argument for this integration is that it owns every layer of the stack — custom TPU silicon, frontier Gemini models, the cloud platform itself, and a large enterprise distribution channel through Workspace — so a single platform surface is both possible and preferable to splitting concerns the way Azure and AWS do.

<Callout type="note" title="A moving target">
  As of mid-2026, Google has been repositioning Vertex AI's branding and roadmap under a broader "Gemini Enterprise" agent-platform umbrella, reflecting the industry-wide shift toward agentic workflows as the primary application pattern. The underlying architectural building blocks described above — Model Garden, custom training, model registry, endpoints — continue to function essentially as before; treat product naming as more volatile than the architecture itself, and verify current naming against Google Cloud's official documentation before committing to it in a design doc.
</Callout>

## Comparing the Three Major Clouds' ML Philosophy

<ComparisonTable
  headers={["Cloud", "How build-vs-consume is split", "Where custom training happens", "Where hosted foundation models live"]}

  ]}

No split is objectively superior — each reflects a different bet about how teams are organized. Separate products (AWS, Azure) let specialized teams (classic ML engineers vs. application/LLM engineers) work in tools scoped to their concerns without wading through irrelevant surface area. A unified platform (Google) reduces the friction of a workload that needs both a custom-trained component and a foundation-model component, since they already share tooling, IAM, and monitoring.

## A Minimal Example of Each

```python
# Azure AI Foundry: call a hosted foundation model from the catalog
from azure.ai.projects import AIProjectClient
from azure.identity import DefaultAzureCredential

project = AIProjectClient.from_connection_string(
    credential=DefaultAzureCredential(),
    conn_str="<foundry-project-connection-string>",
)
client = project.inference.get_chat_completions_client()
response = client.complete(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Summarize this support ticket: ..."}],
)
```

```python
# Azure Machine Learning: submit a custom training job
from azure.ai.ml import MLClient, command
from azure.identity import DefaultAzureCredential

ml_client = MLClient.from_config(credential=DefaultAzureCredential())

job = command(
    code="./src",
    command="python train.py --n_estimators 200 --data ${{inputs.data}}",
    inputs={"data": "azureml://datastores/workspaceblobstore/paths/training-data/"},
    environment="azureml://registries/azureml/environments/sklearn-1.5/versions/1",
    compute="cpu-cluster",
)
ml_client.jobs.create_or_update(job)
```

```python
# Vertex AI: one platform for both a hosted model call and custom training
from google.cloud import aiplatform

aiplatform.init(project="my-project", location="us-central1")

# Call a hosted foundation model from Model Garden
model = aiplatform.gapic.PredictionServiceClient()

# ...or launch custom training on the same platform
job = aiplatform.CustomTrainingJob(
    display_name="fraud-model-training",
    script_path="train.py",
    container_uri="us-docker.pkg.dev/vertex-ai/training/scikit-learn-cpu.1-0:latest",
)
model = job.run(
    replica_count=1,
    machine_type="n1-standard-4",
    dataset=my_dataset,
)
endpoint = model.deploy(machine_type="n1-standard-4")
```
