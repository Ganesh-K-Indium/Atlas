## Intuition

Most ML tooling assumes your data and computation fit on one machine. Once a dataset is too large for that — hundreds of gigabytes to petabytes — you need a system that splits both the data and the computation across many machines, coordinates them, and recovers automatically when one of those machines fails. **Apache Spark** is the dominant engine for this, and **Databricks** is the company founded by Spark's original creators that turned it into a managed platform, then layered a data-warehouse-style storage architecture (the **Lakehouse**) and an ML/AI product suite on top.

<Callout type="tip" title="The one-sentence idea">
  Spark solves "how do I run one logical computation across a cluster of machines without writing distributed-systems code myself"; Databricks solves "how do I get a managed Spark cluster, a governed table format that behaves like a warehouse but stores data like a data lake, and an ML/LLMOps platform, without operating any of that infrastructure myself."
</Callout>

## Spark's Core Distributed-Computing Model

Spark's foundational abstraction, introduced in the original Zaharia et al. paper, was the **Resilient Distributed Dataset (RDD)**: an immutable, partitioned collection of records distributed across a cluster, with enough lineage information (the sequence of transformations that produced it) to recompute any lost partition after a machine failure — this is what "resilient" means. Modern Spark code mostly operates on the higher-level **DataFrame** API, which is built on RDDs but adds a schema and lets Spark's query optimizer (Catalyst) plan the computation more intelligently, closer to how a SQL engine plans a query.

Three ideas make Spark's model precise:

- **Driver-executor architecture:** a single **driver** process holds the logical plan and coordinates work; many **executor** processes, each running on a cluster node, actually process data partitions in parallel. The driver never touches the raw data at full scale — it only ships instructions and collects small results (aggregates, samples) back.
- **Partitioning:** data is split into partitions, the unit of parallelism. Operations that only need each partition's own data (`map`, `filter`) run embarrassingly parallel with no cross-machine communication. Operations that need data grouped by some key across partitions (`groupBy`, `join`) require a **shuffle** — physically moving data between executors over the network — which is by far the most expensive operation in Spark and the usual target of performance tuning.
- **Lazy evaluation:** transformations (`map`, `filter`, `select`, `join`) build up a logical plan but execute nothing. Only an **action** (`count`, `collect`, `write`) triggers Catalyst to optimize the accumulated plan and actually run it. This lets Spark fuse and reorder operations (e.g. push a filter before a join) for a much cheaper physical execution than a naive line-by-line interpretation would produce.

## Databricks: Lakehouse and ML Platform on Top of Spark

Databricks' central architectural bet is the **Lakehouse**: store data in open file/table formats (built on Parquet, with transactional guarantees via Delta Lake) directly in cheap cloud object storage — the economics and flexibility of a data lake — while adding the schema enforcement, ACID transactions, and governance (via **Unity Catalog**) traditionally associated with a data warehouse. This removes the older two-system pattern of a data lake for raw/ML data plus a separate warehouse for BI, which required constantly copying and reconciling data between them.

On top of that foundation, Databricks' **Mosaic AI** is the current umbrella for its ML/AI-specific capabilities:

- **MLflow**, an open-source project that originated at Databricks, is the de facto standard for experiment tracking, model packaging, and model registry across the industry — not limited to Databricks itself. **MLflow 3** extends this into GenAI-specific concerns: tracing agent/LLM calls and storing them (increasingly in OpenTelemetry format, governed through Unity Catalog table permissions) for evaluation and debugging.
- **Model Serving** hosts both custom-trained models and foundation models behind managed endpoints, with an **AI Gateway** in front for centralized governance, rate limiting, and monitoring of access to those endpoints.
- **Vector Search** is a serverless vector database integrated with the Lakehouse, used to build RAG pipelines directly against governed enterprise data — see [Retrieval-Augmented Generation (RAG)](/docs/retrieval-knowledge/rag/retrieval-augmented-generation) for the retrieval mechanics this implements.
- **Agent Bricks / Agent Framework** provide managed tooling for building, evaluating, and deploying LLM-based agents, including automatic generation of domain-specific evaluation benchmarks — an area Databricks has invested in heavily as agentic workflows have become the dominant new application pattern industry-wide.

The unifying theme: Databricks' pitch is that data engineering (Spark, Delta Lake), data warehousing/BI, and ML/AI development can all run against the *same* governed copy of data in the *same* platform, rather than requiring separate systems that need to be kept in sync.

## A Minimal Example

```python
# PySpark: lazy transformations, then an action triggers execution
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, avg

spark = SparkSession.builder.appName("feature-engineering").getOrCreate()

# Nothing executes yet — this just builds a logical plan
transactions = spark.read.table("catalog.schema.transactions")
enriched = (
    transactions
    .filter(col("amount") > 0)
    .groupBy("customer_id")
    .agg(avg("amount").alias("avg_transaction_amount"))
)

# .count() is an action — this is what actually triggers a shuffle + execution
print(enriched.count())
enriched.write.mode("overwrite").saveAsTable("catalog.schema.customer_features")
```
