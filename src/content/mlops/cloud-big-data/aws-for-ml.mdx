## Intuition

"AWS for ML" is not one product, it's a spectrum, and the two anchor points on that spectrum — **SageMaker** and **Bedrock** — solve genuinely different problems. Confusing them is the single most common architectural mistake teams make when starting on AWS.

**SageMaker** exists for teams that need to control the model itself: training a custom model from scratch or fine-tuning one on proprietary data, choosing the exact compute, and owning the full training-to-serving lifecycle. **Bedrock** exists for teams that don't want to own any of that — they want to call a frontier foundation model (from Anthropic, Meta, Mistral, Amazon's own Nova family, and others) through one HTTP API and let AWS manage the serving infrastructure entirely.

<Callout type="tip" title="The one-sentence idea">
  SageMaker is a managed platform for *building and operating your own model infrastructure* (bring code, own the container, own the endpoint); Bedrock is a managed API for *renting inference against someone else's already-trained foundation model* — the architectural decision is whether your differentiation lives in the model weights or in what you build around a model you don't own.
</Callout>

The two are not competitors within AWS's own lineup — they're complementary, and mature AWS ML architectures typically use both: SageMaker for a proprietary tabular fraud model, Bedrock for the customer-facing chat layer sitting on top of it.

### SageMaker: own the training-to-serving lifecycle

SageMaker is best understood as *managed infrastructure for the classic ML lifecycle*, not a single tool:

- **Training jobs** run your training script (or a built-in algorithm) on managed compute you select, including **SageMaker HyperPod**, which provisions resilient clusters designed to automatically recover from hardware failures during long, expensive distributed training runs — a real pain point once training jobs span many days across hundreds of GPUs.
- **Inference** supports multiple serving patterns: real-time endpoints for low-latency synchronous requests, serverless endpoints for spiky/intermittent traffic, batch transform for offline scoring, and asynchronous endpoints for large payloads with relaxed latency needs. An **inference pipeline** chains up to fifteen containers (preprocessing → model → postprocessing) behind a single endpoint.
- **SageMaker Pipelines** orchestrates the full lifecycle — data prep, training, evaluation, conditional deployment — as a versioned, auditable CI/CD workflow, authorable via SDK, API, or a visual drag-and-drop editor in SageMaker Studio.
- **Model Monitor** watches deployed endpoints for data drift, concept drift, and bias, closing the loop between "we shipped a model" and "we know when it's degrading."

The unifying theme: every step is a managed AWS service, but you still own the model artifact, the training code, and the decision of what gets deployed. That control is the point — and the cost.

### Bedrock: rent inference on models you don't own

Bedrock's core promise is a **single unified API** in front of foundation models from multiple providers, with no infrastructure to provision and no minimum commitment. As of 2026, the catalog spans text, vision, audio, and code models from Anthropic, Meta, Mistral, Amazon's own Nova family, and several other providers, reflecting a shift from Bedrock's original text-first scope toward a broadly multimodal one.

Around the raw `InvokeModel` call, Bedrock layers managed application-building blocks so teams don't have to hand-roll them:

- **Knowledge Bases** — a managed RAG pipeline: point it at a data source (e.g. S3), and Bedrock handles chunking, embedding, vector storage, and retrieval-augmented prompting, without your team standing up a vector database. See [Retrieval-Augmented Generation (RAG)](/docs/retrieval-knowledge/rag/retrieval-augmented-generation) for the underlying mechanics this replaces.
- **Agents / AgentCore** — managed orchestration for tool-calling and multi-step agentic workflows against a chosen foundation model.
- **Guardrails** — configurable content-filtering and hallucination-mitigation policies applied consistently across models and applications.
- **Custom Model Import / fine-tuning** — for teams that want a foundation model adapted to their domain without owning general-purpose training infrastructure, Bedrock supports importing custom weights or fine-tuning selected models while keeping the same serverless invocation API.

The trade-off for all this convenience is reduced control: you don't choose the underlying training data, you're bound by the provider's rate limits and model versioning cadence, and deep architectural changes to the model itself are simply not available to you.

## A Minimal Example of Each

```python
# SageMaker: launch a training job and deploy a custom model
import sagemaker
from sagemaker.sklearn.estimator import SKLearn

session = sagemaker.Session()
role = "arn:aws:iam::123456789012:role/SageMakerExecutionRole"

estimator = SKLearn(
    entry_point="train.py",
    role=role,
    instance_type="ml.m5.xlarge",
    framework_version="1.2-1",
    hyperparameters={"n_estimators": 200, "max_depth": 8},
)
estimator.fit({"train": "s3://my-bucket/training-data/"})

predictor = estimator.deploy(
    initial_instance_count=1,
    instance_type="ml.m5.large",
)
result = predictor.predict(sample_features)
```

```python
# Bedrock: call a hosted foundation model, no infrastructure owned
import boto3
import json

bedrock = boto3.client("bedrock-runtime", region_name="us-east-1")

response = bedrock.invoke_model(
    modelId="anthropic.claude-sonnet-4-5-20250929-v1:0",
    body=json.dumps({
        "anthropic_version": "bedrock-2023-05-31",
        "max_tokens": 512,
        "messages": [{"role": "user", "content": "Summarize this incident report: ..."}],
    }),
)
output = json.loads(response["body"].read())
```
