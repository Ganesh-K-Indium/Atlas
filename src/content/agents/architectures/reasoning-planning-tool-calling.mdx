## Intuition

A language model that only produces text is a very capable autocomplete. An *agent* is that same model wired into a loop where it can also act — query a database, call an API, run code — observe what happened, and decide what to do next. The interesting design question isn't "can the model call a tool," it's **how the model's reasoning and its actions are sequenced**: does it think through the whole plan up front, or does it think a little, act a little, and let real observations steer the next thought?

<Callout type="tip" title="The one-sentence idea">
  ReAct interleaves reasoning and acting one step at a time so each action is informed by the latest observation; plan-and-execute frameworks commit to a multi-step plan first and only revise it when execution reveals a problem. Both ultimately rely on the same underlying mechanism — structured tool calling — to actually touch the world.
</Callout>

## The ReAct Pattern

ReAct ("**Rea**soning + **Act**ing"), introduced by Yao et al. (2022), prompts the model to produce an explicit interleaved trace: a **Thought** (free-text reasoning about what to do next), an **Action** (a tool call, formatted so the host can parse and execute it), an **Observation** (the tool's result, inserted back into context), and then another Thought — repeating until the model emits a final answer.

$$
\text{Thought}_1 \to \text{Action}_1 \to \text{Observation}_1 \to \text{Thought}_2 \to \text{Action}_2 \to \text{Observation}_2 \to \cdots \to \text{Answer}
$$

The key property is that reasoning and acting are **mutually grounding**: the reasoning trace helps the model decide which action makes sense, and the observation from each action gives the next reasoning step something real to condition on — rather than the model hallucinating what a tool's result "probably" would be. This tends to reduce compounding errors compared to chain-of-thought alone, because the model can't drift far off course without a real observation checking it at every step.

```python
# Minimal from-scratch ReAct loop sketch (illustrative, not a specific framework's API)
def react_loop(llm, tools: dict, question: str, max_steps: int = 6) -> str:
    scratchpad = f"Question: {question}\n"
    for _ in range(max_steps):
        response = llm.generate(
            prompt=REACT_PROMPT_TEMPLATE.format(scratchpad=scratchpad),
            stop=["Observation:"],
        )
        scratchpad += response  # appends "Thought: ...\nAction: tool_name[args]"

        if "Final Answer:" in response:
            return response.split("Final Answer:")[-1].strip()

        tool_name, args = parse_action(response)
        observation = tools[tool_name](args)
        scratchpad += f"Observation: {observation}\n"

    return "Max steps reached without a final answer."
```

## Plan-and-Execute: Committing to a Plan Upfront

An alternative architecture separates planning from execution into two distinct phases. A **planner** call (often a stronger or more carefully prompted model) first decomposes the task into an explicit multi-step plan — "1. Look up X, 2. Compute Y from X, 3. Compare Y to Z" — before any tool has been touched. An **executor** then works through the plan step by step, optionally re-planning if a step's result invalidates a later one.

Plan-and-execute trades away ReAct's step-by-step responsiveness for a plan the developer (and the model) can inspect, critique, or edit before any side-effecting action runs — valuable when actions are expensive, irreversible, or need a human sign-off before execution. It also lets the executor be a cheaper/faster model than the planner, since executing an already-decided step needs less reasoning than deciding what to do next from scratch.

## Tool-Calling Mechanics

Both patterns above rely on the same underlying capability: the model needs a reliable way to invoke a function and receive its result back into context. Modern tool calling works like this:

1. **Schema declaration**: the host describes each available tool with a name, natural-language description, and a typed argument schema (usually JSON Schema) — this is passed to the model alongside the conversation, not baked into the prompt as free text.
2. **Structured emission**: instead of writing prose, the model emits a structured call (e.g. a JSON object naming the tool and its arguments) when it decides a tool is needed. Providers typically train models specifically to emit this format reliably and to know when *not* to call a tool.
3. **Host-side execution**: the *host application*, not the model, actually executes the call — running the code, hitting the API, querying the database. The model never directly touches the outside world; it only ever emits an intent to act.
4. **Result injection**: the tool's return value is serialized and inserted back into the conversation as an observation, and generation continues from there.

This separation of "model decides/describes" from "host executes" is what makes tool calling safe to gate: a host can validate arguments, ask for human confirmation on risky calls, or simply refuse to run a requested tool, all without the model having any lower-level access itself. It's also the same execution boundary that protocols like [MCP](/docs/agents/protocols-frameworks/model-context-protocol) standardize at scale — MCP doesn't change this basic mechanic, it standardizes how the tool schemas are discovered and where the host sends the call.

## Common Mistakes

- **Treating the reasoning trace as free**: every Thought is still tokens the model generates and the host pays for and re-sends each turn; unbounded reasoning traces blow up context and latency.
- **No loop termination guarantee**: a ReAct loop with no max-step cap or explicit "give up" condition can spin indefinitely if the model never emits a final answer — always cap steps and handle the failure path.
- **Assuming plan-and-execute plans are static**: real-world tasks often invalidate step 3 of a plan by the time step 1 finishes; a plan-and-execute system with no re-planning trigger will happily execute a stale plan.
- **Conflating "the model called a tool" with "the tool ran safely"**: without host-side argument validation and, for risky actions, human confirmation, structured tool calling is just as capable of causing damage as any other code path — the structure doesn't imply safety.
