## Intuition

MCP solves how one agent talks to a *tool*. A2A solves a related but distinct problem: how one autonomous agent talks to *another autonomous agent* — potentially built by a different team, on a different framework, with no shared code and no visibility into the other's internal reasoning. Without a shared protocol, every pair of interoperating agents needs its own bespoke integration, the same $M \times N$ problem MCP solved for agent-to-tool, just one layer up.

<Callout type="tip" title="The one-sentence idea">
  A2A is an open protocol for agent-to-agent interoperability: agents publish "Agent Cards" describing their capabilities so other agents can discover them, then communicate through a standardized task lifecycle (submit a task, get status updates, exchange artifacts) without either side needing to know how the other is implemented internally.
</Callout>

## Origin and Governance

A2A was introduced by Google in April 2025 as an open protocol for cross-vendor agent interoperability. On June 23, 2025, Google donated A2A to the **Linux Foundation**, establishing it as an independently-governed Agent2Agent Protocol Project rather than a single-vendor standard — mirroring the same move toward neutral, multi-vendor stewardship that MCP made under the Agentic AI Foundation.

<Callout type="warning" title="Fast-moving governance and adoption numbers">
  As of this page's last-verified date, A2A has reported reaching v1.0 with signed Agent Cards and an extension mechanism (v1.0.1), 22,000+ GitHub stars, SDKs across five languages (Python, JavaScript, Java, Go, .NET), and 150+ organizations reported in production. Treat specific adoption figures and version numbers as a snapshot — check the Linux Foundation's Agent2Agent Protocol project page and the official spec for current status before building against it.
</Callout>

## A2A vs. MCP

These two protocols are frequently confused because both are "agent protocols," but they standardize different relationships:

<ComparisonTable
  headers={["", "MCP", "A2A"]}
  rows={[
    ["Relationship standardized", "Agent ↔ tool/data server", "Agent ↔ agent"],
    ["Typical unit exposed", "Tools, Resources, Prompts (a server's capabilities)", "An Agent Card describing a whole agent's skills and how to invoke it"],
    ["Interaction shape", "Discrete function calls with structured arguments and returns", "A task lifecycle — submit, track status over time, receive artifacts, potentially over a longer-running interaction"],
    ["Governance", "Agentic AI Foundation (Linux Foundation directed fund), donated by Anthropic Dec 2025", "Agent2Agent Protocol Project (Linux Foundation), donated by Google June 2025"],
    ["Complementary, not competing", "An agent still needs MCP to call its own tools", "The same agent can expose itself to *other* agents via A2A while internally using MCP to reach its tools"],
  ]}
/>

The two are designed to compose: an agent built with MCP-based tool access can itself be exposed as an A2A-discoverable agent to other systems, and a multi-agent orchestrator (see [Multi-Agent Systems](/docs/agents/architectures/multi-agent-systems)) may use A2A to delegate a sub-task to an independently-hosted agent while that agent, internally, uses MCP to reach its own tools and data. See [Model Context Protocol (MCP)](/docs/agents/protocols-frameworks/model-context-protocol) for the agent-to-tool side of this layering.

## A Minimal Agent Card Sketch

```json
{
  "name": "invoice-processing-agent",
  "description": "Extracts structured line items from invoice documents and validates them against a purchase order.",
  "url": "https://agents.example.com/invoice-processor",
  "version": "1.0.0",
  "capabilities": {
    "streaming": true,
    "pushNotifications": false
  },
  "skills": [
    {
      "id": "extract-invoice",
      "description": "Extract structured line items, totals, and vendor info from an invoice document.",
      "inputModes": ["file/pdf", "text/plain"],
      "outputModes": ["application/json"]
    }
  ]
}
```

A calling agent fetches this card, confirms the `extract-invoice` skill matches what it needs, and submits a task referencing that skill — without ever needing to know whether the invoice-processing agent is built on LangGraph, CrewAI, or a custom stack.

## Common Mistakes

- **Assuming A2A replaces MCP**: they address different relationships; an agent that adopts A2A for external interoperability still needs a way (often MCP) to reach its own tools internally.
- **Treating an Agent Card as a guarantee of behavior**: a card describes advertised capabilities, not a formal contract on output quality or correctness — the calling agent still needs its own validation of returned artifacts.
- **Ignoring the task lifecycle's intermediate states**: treating a delegated A2A task as a single synchronous call rather than tracking its status transitions can cause a calling agent to time out on or mishandle legitimately long-running remote work.
- **Building against unstable specifics without checking current governance**: A2A moved organizations once already (Google to Linux Foundation) and continues to version its spec — cite the current spec source rather than a cached understanding when accuracy matters.
