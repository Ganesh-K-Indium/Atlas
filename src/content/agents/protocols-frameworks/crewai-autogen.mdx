## Intuition

If LangGraph's pitch is "model your agent's control flow as an explicit graph," CrewAI and AutoGen each start from a different mental model for *multi-agent* systems specifically. CrewAI's is organizational: define agents with roles, goals, and tools, like staffing a team. AutoGen's is conversational: agents are participants in a structured multi-turn conversation, and the conversation pattern itself (who talks to whom, in what order) is the main thing you configure.

<Callout type="tip" title="The one-sentence idea">
  CrewAI structures multi-agent collaboration around explicit roles (a "Researcher," a "Writer," a "Reviewer," each with a goal and tools) working together as a crew; AutoGen structures it as configurable multi-turn conversations between agents. Neither is strictly better — they encode different intuitions about what a "team" of agents should look like.
</Callout>

## CrewAI: Role-Based Collaboration

In CrewAI, you define each agent with a role, a goal, an optional backstory (used to steer the model's persona/behavior), and a set of tools — then group agents into a **Crew** that works through a shared task list, with CrewAI managing hand-offs between them. CrewAI also has **Flows**, a separate event-driven layer for precise, deterministic workflow control (single LLM calls, explicit branching) that can incorporate a Crew as one step rather than relying purely on agent-to-agent autonomy for the whole pipeline.

```python
# CrewAI-style role-based crew (illustrative)
from crewai import Agent, Task, Crew

researcher = Agent(
    role="Senior Researcher",
    goal="Find and summarize the most recent developments in a given topic",
    backstory="An analyst who values primary sources over secondhand summaries.",
    tools=[web_search_tool],
)

writer = Agent(
    role="Technical Writer",
    goal="Turn research findings into a clear, well-structured brief",
    backstory="A writer who prioritizes precision over flourish.",
)

research_task = Task(
    description="Research recent developments in {topic} and list key findings with sources.",
    agent=researcher,
    expected_output="A bulleted list of findings, each with a source.",
)

writing_task = Task(
    description="Write a one-page brief based on the research findings.",
    agent=writer,
    expected_output="A one-page markdown brief.",
    context=[research_task],
)

crew = Crew(agents=[researcher, writer], tasks=[research_task, writing_task])
result = crew.kickoff(inputs={"topic": "agent interoperability protocols"})
```

The role/goal/backstory framing is meant to make each agent's scope legible at a glance — closer to writing a job description than configuring a state machine — which tends to make CrewAI approachable for teams composing "a team of specialists" without wanting to design an explicit graph by hand.

## AutoGen: Conversational Multi-Agent Turns

AutoGen's abstraction is a structured conversation: agents (an "assistant" agent, a "user proxy" agent that can execute code or represent a human, additional specialized agents) exchange messages according to a configurable conversation pattern — two-agent chat, group chat with a manager deciding speaking order, or nested conversations. The conversation *is* the orchestration mechanism: an agent's turn ends when it produces a message, and the pattern determines who speaks next.

```python
# AutoGen-style conversational multi-agent pattern (illustrative)
from autogen import AssistantAgent, UserProxyAgent

assistant = AssistantAgent(
    name="assistant",
    llm_config={"model": "gpt-4o"},
)

user_proxy = UserProxyAgent(
    name="user_proxy",
    code_execution_config={"work_dir": "coding"},
    human_input_mode="NEVER",
)

user_proxy.initiate_chat(
    assistant,
    message="Write and run a script that fetches the top 5 posts from a given subreddit.",
)
# The two agents exchange turns: assistant proposes code, user_proxy executes it
# and returns output/errors, assistant revises, until the task is done.
```

<Callout type="warning" title="What 'AutoGen' means today has changed">
  As of this page's last-verified date, the AutoGen project has split into three distinct paths, and simply saying "AutoGen" is ambiguous without specifying which:
  <br /><br />
  1. **Original AutoGen** is reported to be in maintenance mode — community-managed, no significant new features, still usable for research/prototyping on its existing (v0.4+) async actor-model architecture.
  <br /><br />
  2. **AG2** (at `ag2ai/ag2` on GitHub) is a community-led fork by AutoGen's original paper authors, positioned as an independently-governed continuation under an open-source license, with its own newer API surface under active development.
  <br /><br />
  3. **Microsoft Agent Framework** is Microsoft's own successor, released as 1.0 in April 2026, explicitly described by Microsoft as combining Semantic Kernel's enterprise plumbing with AutoGen's multi-agent conversation patterns — positioned as the production path forward from both predecessor projects.
  <br /><br />
  New projects should evaluate AG2 or Microsoft Agent Framework rather than building fresh on the original AutoGen — check each project's current documentation before committing, since this landscape has shifted more than once in a short period.
</Callout>

## Common Mistakes

- **Assuming "AutoGen" still means one thing**: given the maintenance-mode/AG2/Microsoft Agent Framework split, code, tutorials, or job postings referencing "AutoGen" may target any of three different, diverging projects — confirm which before reusing a pattern.
- **Over-relying on emergent conversation patterns for tasks needing determinism**: a group-chat-style pattern that lets a manager agent decide turn order can produce inconsistent execution paths across runs; tasks needing predictable, auditable execution often fit better with an explicit graph (LangGraph) or a deterministic Flow (CrewAI) than a fully emergent conversation.
- **Treating role/goal/backstory prompts as a substitute for tool/schema design**: a well-written role description doesn't compensate for poorly specified tools or ambiguous task boundaries between agents — the same tool-calling and hand-off discipline from [Multi-Agent Systems](/docs/agents/architectures/multi-agent-systems) still applies.
- **Picking a framework by popularity alone**: framework adoption and positioning are moving targets in this space; validate against the current project docs and your own coordination needs (deterministic vs. conversational, in-house vs. cross-organization) rather than defaulting to whichever is trending.
